14分布式数据实时管理技术

基本概念

这个项目 4 个研究内容。
5 个指标内容：

  1. 支持不少于100个分布式节点多模态海量异构数据实时管理；
    1. 100 个以上的节点；
  2. 支持跨平台多模态数据检索的数据类型包括但不限于数值型、图片、文本等类型；
    1. 传输的数据量达到 10 Mb 至少；
  3. 支持数据挖掘内容包括但不限于飞行状态、战场环境和目标等数据；
  4. 全局节点并发跨平台查询吞吐量不低于1百万QPS；
    1. 并发查询请求数量≥10个；
    2. 并发涉及的节点数量≥10个；
  5. 支持节点间态势信息主动推送能力；
    1. 态势信息种类≥2个，包括但不限于目标类型、目标飞行状态信息；
    2. 支持基于订阅条件的推送和智能推送能力；

研究内容是由一个主架构、两个关键技术(一个分布式检索、一个分布式推送)、一个演示验证系统(分为服务端和客户端)四个部分构成。
概念词是多源异构，那么在设计的时候只需要对演示验证系统，抽象模拟出多个异构的对象，进行不同的信息输入即可。
两个关键技术是比较重要的。主要的技术概念都在这两个关键技术，分布式检索中提到使用向量数据库；分布式推送旨在进行查询，然后推送。实际上分布式推送相当于拆成了两个部分，一个是查询后对结果的反馈，一个是基础的推送工具。

所以，分布式的内容不仅仅只是网络分布式，也包括进行分布式存储，分布式检索。

应用场景

作战、军事化、分布式应用场景。
调研基准：
涵盖一个空想式的大的场景，提出若干个更加细节的场景，其中细节场景需要论文佐证。

构建一个全面信息化的战场环境并不是一件容易的事情。
理论上来说，我们可以对战场中的所有电子设备都负载一个数据处理元件，其具有相应的感知功能，允许本地先进行缓存，加入组网后将数据上传。
如果距离允许，且设备负载了电源，可以充当一个提供服务的节点，在有限空间域范围内接收所有节点的信息————这就表明软件的研制依赖一个可扩展的层次网络结构，使得其能够对一部分节点的网络视为一个节点内容。
设备作为采集元件，将数据上报，至少是其位置信息。
在此基础上，战场的信息构建完成后，能够做的事情将是指数上升。作为全面信息化的作战部署，其能够应用在包括指挥控制系统上，调度所有的信息设备并且下发指令。
甚至能够直接指挥导弹发射和人员作战。
这些都单属于信息化的范畴。

如果论调亮点，那么针对多模态的数据进行适应，是信息化中支持大量级、多异构数据的处理是其一大亮点。
抽象出计算节点、基础采集节点、服务节点、主服务集群、指控端这些角色。

全面场景模型

在信息化战场中，每台电子设备均集成了一个具备数据处理与感知能力的智能单元。这些单元不仅能够感知环境数据（如位置、状态、周边信号强度等），还可在本地存储和处理数据，并在接入网络后，将重要信息上传至指挥中心或相邻节点。这种信息化生态构成了战场动态信息采集与实时共享的基础。

场景背景

在一个复杂地形（如山地或城市区域）的战场中，传统通信方式(基于固定基础设施的通信、点对点无线电通信、基于集中式结构的通信网络、卫星通信)受环境影响较大，存在信号遮挡和信息延迟问题。为了克服这些挑战，构建围绕空中设备的分布式网络系统。该系统由大量节点组成，包括士兵携带的智能终端、无人机平台、地面机器人和固定设施上的通信设备。

节点功能描述
  1. 基础采集节点（士兵终端/传感器）
    1. 功能：采集位置信息、周边环境数据（温度、湿度等）及战场态势（敌方信号、目标动态）。
    2. 任务：本地缓存数据，优先上传关键信息（如敌方坐标），并接受来自指挥中心的命令。
  2. 中继服务节点（无人机/地面机器人）
    1. 功能：作为移动服务节点，在信号覆盖受限区域内提供通信中继和分布式数据处理能力。
    2. 任务：汇总周边节点上传的数据，筛选关键信息并发送至指挥中心或其他高优先级节点。
  3. 核心指挥节点（指挥中心/战术车辆）
    1. 功能：实时汇总全局数据，生成战场态势图，为指挥官提供决策支持。
    2. 任务：根据接收到的战场信息，对人员和武器系统进行调度，并实时下达指令。
应用流程
  1. 初始组网
    士兵携带的终端和部署在战场的无人机自主组网，通过分层协议将每个设备的位置信息上传至指挥中心。
  2. 态势感知与数据处理
    基础节点实时采集周边环境数据（如热源、振动、声音等），并通过中继节点汇总，构建出战场的多维态势图。
    系统利用智能算法对数据进行优先级分类，将关键信息（如敌方火力点、弹药耗尽的友军位置）推送给相关单位。
  3. 战术部署
    指挥中心基于态势图制定作战方案，并通过网络将指令传达至前线部队和武器系统。
    无人机和地面机器人执行自动化指令，完成目标打击或资源运送任务。
  4. 动态调整
    当战场局势发生变化时，系统通过实时更新的分布式网络重新分配节点任务。
    新接入的设备能够自动加入网络，并根据分层结构迅速开始数据上传和执行任务。
系统优势
  1. 高效性：节点分层设计和本地缓存机制显著降低数据传输延迟，确保指令的实时性。
  2. 鲁棒性：通过多节点冗余设计和自适应网络结构，应对设备故障和信号干扰。
  3. 扩展性：可根据需求灵活增减节点规模，适配不同作战规模和场景。
数据处理

战场中的电子设备作为分布式传感器网络的一部分，不断采集位置信息、环境数据和目标动态。这些数据通过本地预处理后上传至指挥系统，形成以下两类核心数据：

  • 静态数据：包括地形、友军部署位置、基础设施信息等，为战略规划提供参考。
  • 动态数据：包括敌方位置、弹药消耗、实时战损评估等，为战术调整提供依据。
决策优势

在战术层面，实时决策优化是战场信息化的关键目标之一。系统支持内容：

  1. 资源分配：
    根据敌我态势，动态调整武器装备、后勤物资和支援力量的分配。
    通过实时计算关键节点的任务优先级，确保资源使用效率最大化。
  2. 路径规划：
    利用节点的位置信息和战场地形数据，为无人机、地面部队规划安全高效的行进路线。
    系统根据威胁等级动态调整路径，避免误入高危区域。
  3. 火力分配：
    基于敌方目标优先级和火力覆盖范围，分配适当武器系统（如导弹、无人机、火炮）。
    决策支持系统模拟多种攻击方案，分析效果后推荐最优方案。
    4. 多节点协同：
    在任务执行过程中，系统实时监测各节点状态（如电量、武器状态），自动分配任务。
    若某节点失联或故障，系统可快速重组网络，并将任务重新分配给其他节点。
  4. 决策流程
    感知阶段：分布式节点感知战场情况，完成数据采集与本地分析。
    • 汇总阶段：通过多层网络结构，将各节点的关键信息汇聚到指挥中心。
    • 分析阶段：指挥中心利用大数据分析和人工智能模型（如机器学习、模式识别）生成态势评估和预测结果。
    • 输出阶段：基于分析结果，系统自动生成多套备选方案供指挥官选择，并将指令下发至执行单位。
  5. 决策系统的智能化特点
    • 自适应性：能够根据战场环境变化动态调整优先级，优化决策模型。
    • 预测性：通过态势分析和趋势建模，提前预警潜在威胁并提供解决方案。
    • 高容错性：即使部分节点失效，系统仍能通过冗余设计确保信息完整性和决策可靠性。
    • 人机协作：系统通过可视化界面与指挥官交互，提供分析结果和建议，同时允许指挥官根据经验修正策略。
结论

上述场景体现了未来战场全面信息化的应用潜力。通过感知-处理-决策-执行的闭环流程，该系统显著提升了作战效率与精准性，为现代化军事行动提供了可靠的技术保障。

细节场景

主要偏向于空域中的高实时协同应用。

空射诱饵弹

空射诱饵弹作战样式分析是一个典型的现代化作战协同场景。
在导弹打击过程中，通常由空中单位（如战斗机、无人机）或地面单位（如导弹发射车、战术指挥中心）对导弹的发射进行操控。这一阶段包括目标选择、发射准备以及导弹制导系统的初始化。一旦导弹完成发射，通常会进入自主飞行阶段，利用其内置的制导系统和目标识别技术，独立完成对目标的追踪与打击任务。这是一个独立的过程。
但是单个导弹的打击效果有限且容易被拦截，多个导弹将耗费大量的成本，因此，构建围绕有效打击设施、伴随多种兵器进行协同作战成为现代化空域作战的必然需求。多兵种协同需要满足远程打击，又涉及到多段通信问题。空中编队形成可靠协同，共同侦测破击和降低蓝方防空体系威胁（包括空中预警机、战斗机，地面预警雷达、搜索雷达、制导雷达以及海面舰载防空系统等），形成有效打击。

在模拟编队佯动、诱骗火力消耗、协同防空压制以及末段协同突防等作战场景中，数据的实时管理是支撑整个体系运行的核心技术之一。这项技术不仅为各个作战单元之间的协调提供了高效的数据共享能力，还确保了复杂战场环境下的信息流畅通和决策精准性。无论是动态调整编队位置以迷惑敌方侦察，还是在防空压制中优化火力分配，数据实时管理技术都在背后发挥着不可替代的作用，为整体作战效果提供了强有力的支撑。

基础设备

在空射诱饵弹作战场景中，基础设备构成是整个体系运行的硬件基础，主要包括以下几个方面：

  1. 空中平台：
    战斗机、无人机以及电子战飞机作为主要空中平台，具备较强的部署灵活性和快速响应能力。这些平台不仅能够搭载诱饵弹，还可以作为通信中继节点或电子战实施平台，承担实时指挥、目标侦察和火力支援等任务。
  2. 地面平台：
    包括预警雷达、搜索雷达、导弹发射装置和战术指挥中心等。这些设备负责目标信息的探测与初步处理，并通过数据链与空中平台进行信息交互，为作战单位提供支撑。
  3. 弹载设备：
    空射诱饵弹内部装备的传感器、通讯模块和干扰设备是作战效果的核心保障。例如，采用多模态传感器（雷达、红外等）完成目标特性探测，搭载电子干扰设备进行敌方信号欺骗或压制，利用数据链模块实现与平台之间的实时通信和协同。
数据特征

数据在空射诱饵弹作战场景中呈现以下几个关键特征：

  1. 多源异构性：
    数据来自多个传感器，涵盖雷达信号、红外成像、光电探测、通信信号等不同类型，具有显著的异构特性。为了确保数据的有效利用，需要进行融合处理，以形成统一的目标态势。
  2. 实时性与动态性：
    数据采集和处理必须跟随目标的实时动态变化，特别是在高速移动目标的情况下，要求系统具备毫秒级的响应能力，避免数据滞后影响决策。
  3. 强抗干扰性：
    作战环境中的电子干扰会导致数据丢包、信号失真等问题，系统需具备强大的抗干扰能力和一定的容错机制，确保数据的连续性与可靠性。
  4. 高数据流量与压缩需求：
    作战时产生的大量传感器数据需要通过通信链路实时传输，在有限的带宽条件下，需对数据进行智能压缩，优先传输关键信息，降低传输负担
通信支持

通信是空射诱饵弹与其他作战单位协同的基础，其支持体系包括以下内容：

  1. 多频段通信：
    系统采用多频段通信方式，覆盖高频、超高频和毫米波频段，提升通信的多样性和适应性，适用于不同作战场景。
  2. 抗干扰通信链路：
    使用跳频通信、扩频技术和加密机制，提升链路的抗截获和抗干扰能力，确保信息在强敌干扰环境中的传递稳定性。
  3. 组网能力：
    利用分布式组网技术，构建空中、地面和海上平台的协同通信网络。在多节点动态部署条件下，确保通信链路的灵活调整和可靠性维护。
  4. 低轨卫星支持：
    引入低轨卫星通信系统，实现大范围覆盖，突破传统地面通信的距离限制，为长距离诱饵弹任务提供可靠的数据链保障。
自动化决策

自动化决策在作战中扮演关键角色，通过实时数据分析与智能计算实现以下功能：

  1. 目标优先级排序：
    基于实时数据，系统可通过AI算法对多个目标进行优先级评估，确定主次打击顺序，优化资源分配。
  2. 任务动态调整：
    在作战环境变化（如目标移动或防空威胁增强）时，系统能够依据最新态势，动态调整任务规划，保持行动的有效性和灵活性。
  3. 协同任务分配：
    自动化决策系统根据平台任务能力和目标分布情况，生成最优任务分配方案，确保多平台协同作战时效益最大化。
  4. 态势预测与行动规划：
    系统通过机器学习模型对目标行为进行预测，并基于预测结果生成行动规划，如调整诱饵弹的轨迹以迷惑敌方雷达，提高整体战术效果。

场景模型

  1. 支撑DOODA”作战环
    “OODA”作战环 （Observe，Orient，Decide andAct） 是由观察、判断、决策和行动构成的作战循环链。首先，传感器探测到敌方目标，并将探测信息传递给指挥决策中心；然后，指挥决策中心对探测信息进行处理，获得态势信息，接着依据战场态势进行决策；最后，下发指令控制已方作战单元展开相关行动。
  2. 电子战平台的“分布式”发展
    电子战飞机可分为预警机、电子干扰机、电子侦察机和防空压制飞机，一架电子战飞机的核心设备由雷达预警和干扰设备、通讯接收机和干扰系统、电子对抗吊舱或其他吊舱等组成。空射诱饵弹可携带侦察、通讯、干扰等不同类型任务载荷，形成分布式电子战平台，单平台根据总体任务目标和本平台的作战能力，通过组网协同方式完成电子战资源的动态调配和优势整合，实施大区域跨度的协同电子战，实现战场“1+1>2”的涌现效应。

设计模块的作用是什么？
确定系统的边界： 设计模块帮助我们确定系统各部分的边界，明确每个模块的责任。例如，数据存储、事务管理、网络通信、故障转移等模块，它们之间有明确的职责划分。
模块化功能划分： 设计模块让我们从整体上思考系统的功能，并将这些功能拆解为具体的模块。每个模块专注于一个独立的职责，从而降低了系统的复杂性。
模块之间的接口和依赖关系： 设计模块时，会考虑到模块间的接口设计和依赖关系。这种抽象的层次能够减少模块之间的紧密耦合，使得未来修改和扩展时更加灵活。
预见潜在问题： 在设计阶段，开发人员可以更早地识别潜在的问题，并对可能出现的难点进行规划。例如，如何保证系统的高可用性、如何处理负载均衡、如何保证数据一致性等问题，都需要在设计时仔细考虑。

TiDB

架构特点支持表级分布式存储
TiDB 采用计算存储分离的架构，主要由 TiDB Server、TiKV 和 PD（Placement Driver）三个核心组件构成，这种架构为表级分布式存储提供了基础。
TiDB Server：作为 SQL 层，负责接收 SQL 请求，进行 SQL 解析、优化和执行计划生成，它本身并不存储数据，而是将数据读写请求转发给存储节点 TiKV。不同表的查询请求都由它来统一处理和调度，为表级数据的分布式访问提供了入口。
TiKV：是分布式键值存储引擎，数据以 Region（一种数据分片）的形式存储在多个 TiKV 节点上。对于数据库中的每个表，会根据一定规则（如范围或哈希）将其数据划分为多个 Region，这些 Region 可以分布在不同的 TiKV 节点上，从而实现表级的分布式存储。
PD：负责整个集群的元数据管理和调度，包括 Region 的分配和负载均衡。它知道每个 Region 所在的位置，当 TiDB Server 需要访问某个表的数据时，会向 PD 询问该表对应 Region 的位置信息，然后再去相应的 TiKV 节点上读写数据。PD 会根据集群的负载情况，动态调整 Region 的分布，确保各个节点的负载均衡。

表级分布式存储的实现方式
数据分片：TiDB 会将表的数据按照一定规则进行分片，默认使用范围分区，也支持哈希分区等方式。例如，对于一个用户表，可以按照用户 ID 的范围将数据划分为多个 Region，每个 Region 存储一定范围内用户 ID 的数据。这些 Region 会被分散存储到不同的 TiKV 节点上，实现表级的分布式存储。
自动负载均衡：PD 会持续监控各个 TiKV 节点的负载情况和存储容量。当某个节点的负载过高或存储容量不足时，PD 会自动将部分 Region 迁移到其他节点上，以保证整个集群的负载均衡。对于表来说，其数据对应的 Region 可能会因为负载均衡而在不同节点间移动，但这对上层应用是透明的。

优势
高可扩展性：可以通过添加 TiKV 节点来线性扩展存储容量，添加 TiDB Server 节点来提高计算能力。对于大规模的数据存储和高并发的查询场景，能够轻松应对。例如，当业务发展导致某个表的数据量急剧增长时，可以通过增加存储节点来存储该表更多的数据。
强一致性：TiDB 使用 Raft 协议保证数据在多个副本之间的一致性。对于表中的数据，无论存储在哪个节点上，都能保证数据的强一致性，确保用户读取到的数据是最新的。
兼容 MySQL：TiDB 兼容 MySQL 协议和大部分语法，这使得用户可以很方便地将原有的 MySQL 应用迁移到 TiDB 上。对于已经使用 MySQL 存储数据的用户，可以平滑过渡到分布式的 TiDB 环境，同时利用其表级分布式存储的优势。

分表的优缺点
优点：
性能提升：分表可以显著减少单表的数据量，从而提高查询和更新性能。
易于扩展：分表让系统更容易横向扩展，可以在多个数据库节点之间分配不同的表。
管理更易：多个小表的管理通常比一个大表简单，数据操作更灵活。

缺点：
跨表查询复杂：如果需要对多个表进行联合查询，查询逻辑会变得更加复杂，可能需要额外的聚合操作或跨表查询。
增加运维成本：如果分表规则没有很好地设计，可能会造成一些表的数据不均衡，增加了后期维护的难度。
开发复杂度：分表通常需要在应用层处理路由，增加了应用开发的复杂度。

数据库 支持分表 支持分片 存储位置 描述
MySQL 是 否 同一数据库实例中的多个表 使用手动分表，通过按范围、哈希、列表等策略分割数据。分表后的数据存储在同一数据库实例的不同表中。
PostgreSQL 是 否 同一数据库实例中的多个表 支持分区表，通过继承表等方式进行分表，分表的数据存储在同一实例中的多个表里。
SQLite 是 否 不同的数据库文件 每个分表可以存储在不同的 SQLite 数据库文件中，适用于轻量级应用。
ClickHouse 是 否 同一数据库实例中的多个表 使用分区表的方式进行分表，分表后的数据仍然存储在同一个节点上，不会跨节点分片。
TiDB 是 是 多个数据库实例或节点 支持自动分表和分片，数据根据分片规则分布到不同的节点，适用于大规模分布式部署。
CockroachDB 是 是 多个数据库实例或节点 支持按范围和哈希分片，可以自动分配表的数据到不同的节点，支持跨节点分布式存储。
MariaDB 是 否 同一数据库实例中的多个表 类似 MySQL，支持通过分区表进行分表，分表的数据仍然存储在同一个数据库实例中的不同表里。
MongoDB 是 是 多个数据库实例或节点 支持自动分片和分表，数据按分片键分布到多个节点，适合横向扩展的大规模数据存储。
Cassandra 否 是 多个数据库实例或节点 Cassandra 是分布式数据库，数据自动根据分片键分配到多个节点，适合大数据量、高并发的场景。
Elasticsearch 否 是 多个数据库实例或节点 数据通过分片和副本进行分布，支持高可用性和扩展性，常用于日志和搜索类应用。

分片策略 行存储的优点 行存储的缺点 列存储的优点 列存储的缺点
哈希分片 - 数据均匀分布，适合 OLTP 场景的高并发写入和读写访问 - 会使某些跨行的查询（如聚合、连接）变得低效 - 对于只需要访问某几列的数据，性能较好 - 跨列查询需要跨多个分片，可能导致性能下降
范围分片 - 适合按时间、ID等有顺序的数据，查询性能较好 - 热点问题，某些分片的负载可能过重 - 对于列式数据，范围查询（如时间范围查询）性能更优 - 数据分布不均，可能导致数据在不同分片间分布不均，出现热点问题
列表分片 - 适合数据比较稀疏且有明确类别时使用 - 可能造成某些类别数据倾斜 - 可根据特定列进行高效查询，针对大数据量的分析查询有优势 - 适应性较差，特别是数据不均匀的情况可能会导致性能下降
复合分片 - 适应性强，能够灵活应对不同的查询需求 - 实现较为复杂，容易带来管理上的困难 - 能更好地应对特定场景（如查询某个特定的列） - 难以管理，尤其在数据量增加时，分片策略可能会导致分片迁移困难

特点 哈希分片 一致性哈希分片
映射方式 将数据通过哈希计算后映射到一个固定的分片上。 将数据映射到哈希环上的节点，节点和数据之间有多个虚拟节点。
扩展性 增加或减少分片时，可能需要重新分配大量数据。 增加或减少节点时，只有部分数据需要迁移，扩展更加平滑。
负载均衡 数据可能因分片键选择不均匀而导致负载不均。 通过虚拟节点机制，能够较好地保持负载均衡。
数据迁移 扩展分片时，可能需要迁移大量数据。 增加或减少节点时，仅有少部分数据需要迁移。
复杂度 实现较简单，主要是通过哈希值和分片数的模运算进行映射。 需要维护哈希环、虚拟节点等，系统实现复杂。
适用场景 适用于节点数相对固定，数据分布较均匀的场景。 适用于动态变化的分布式系统，尤其是分片数量经常变化的场景。
