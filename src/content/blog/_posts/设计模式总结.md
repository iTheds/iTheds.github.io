# 设计模式

各个语言有不同的设计模式。设计模式旨在统一语言的特性结构。本文默认阐述的是基于 C/C++ 语言特性的设计模式。
设计模式众所周知的是[工厂模式等](https://refactoringguru.cn/design-patterns/catalog)，但是仅仅是这些，还未能整体阐述设计模式的基本思想。

设计模式是描述性质启发的，本质是对既定目标的描述。

众所周知，软件开发的过程中，往往是根据应用场景，自上而下的进行落实。
对既定的场景，进行需求分析，构建出可用的需求，这一步往往由市场决定。这个过程即是可行性分析，需求分析。
针对需求，会转化为产品的功能进行描述，但是面向功能的划分方式，不能够作为软件设计。
当然，这要看功能的描述方式是什么。但大部分功能的描述，都是关于产品所应该具有的、面向用户的描述方法。这个过程，实际上还是在需求分析中。
因此， 诸如实现XXX能力, 其表述都不甚准确。

在进行详细设计的过程中，需要将功能转化为详细设计，即对实现功能进行框架的设计。
之后再进行编码、测试。不断重复这个过程，并且使得需求更容易添加迭代。

本文中详细思考关于围绕功能下，如何进行软件设计的方法，所有的软件都应该能通过模型进行描述。

## 基础要义

以两个公理出发，程序在完成之后都是以一套 API 对外进行显示的，程序是运行在最小单位为线程的基础上的。
这两个公理表示了程序的运行方式和结果呈现。

然后在这个基础上，更底层的概念或者说更加广义的概念是程序=数据结构+算法《Algorithms + Data Structures = Programs》。
这表明在设计时，考虑存储和计算方式成为了其关键所在。

按理来说，面向过程和面向对象的编程思想都是可以一齐使用的。
面向对象往往比面向过程更好理解，将动作、实体视为对象，在这个基础上进行多态、封装、继承的特性即可，也符合对事物的描述，对既定行为的描述方式，进行可扩展的学习。

面向过程实际也是在进行面向对象的设计方式。程序对事物的描述，通过一个最小单位的存储结构来表述，只是其无法有效扩展特性，也无法快速进行更迭，而只能通过不断增加过程实现。

实际上，除了面向对象和面向过程还有：
1. 函数式编程
2. 事件驱动编程
3. 面向切面编程
4. 声明式编程
5. 泛型编程
6. 数据流编程
7. 元编程
8. 组件化编程
9. 面向服务编程

但是实际上，这些编程方法是在面向对象的基础上进行更加专有化的操作。而大部分面向对象也只是抽象实体，如果将动作、属性也进行抽象化，就能够对更上层的概念诸如服务、组件视为对象。只是其具有更加专有化的表述方式。

## 面向过程的设计模式

在对面向对象的设计模式进行阐述前，仍然有必要阐述面向过程的设计理念。

面向过程是面向对象的基础，也是最小单位，数据经过流程处理，不断的修正其内容，得到能够使用的结果集合。

## 面向对象的设计模式

### 什么可以抽象为对象？

我们说对象有很多个关系，包括继承、依赖、关联、组合、聚合， 但是实际上，其是没有将对象是什么，已经对象作为什么内容而存在进行完整的、严谨的学术定义。
对象抽象自物理世界的某一个实体，其具有许多的特征，特征不同，进而对象之间的关系不同。
于是，对象的定义，不妨从既定的关系自上而下的推倒出来。
对象体现出多种功能，对象实现一种功能，该功能可以是自底向上的，在不同的功能中，可能依赖多个对象共同协作完成。
单个对象实现特定功能，在此过程中，与其他对象之间的关系形式不同，可以推测出对象关系。

### 对象之间的关系

继承、依赖、关联、组合、聚合。
对象之间的关系决定其影响的方式，也会影响其一系列行为方法。
但是这几种关系难以准确表达。如果引入生命周期的概念，那么会~~一目了然~~:
1. 依赖；实体对象之间的包含关系，被依赖的对象其定义如果改变，那么依赖其的对象的行为也会改变；方法的参数/返回值；对类 B 进行修改会影响到类 A 。
2. 关联；表征一种交互关系，其没有直接的联系，但是通常需要进行单向的交互，持有对其的操作方式；成员，能够访问其成员变量~~友元对象~~；对象 A 知道对象 B。类 A 依赖于类 B。`Teacher ---- Student`
3. 聚合；可有可无~~类内指针~~；对象 A 知道对象 B 且由 B 构成。类 A 依赖于类 B。`Class ◇---- Student`
4. 组合；直接成员成员；对象 A 知道对象 B、由 B 构成而且管理着 B 的生命周期。类 A 依赖于类 B。`Person ◆---- Heart`
其中关系强度为 组合>聚合>关联>依赖。
关联，聚合，组合都是类内的成员。

创建型模式，工厂模式和抽象工厂是最为基础的，即通过继承来实现不同的类定义，只是抽象工厂多一个抽象类。
