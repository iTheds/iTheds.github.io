# TSDB 项目说明文档

## 1. 项目概述

本项目是一个基于 Rust 的时序数据库（TSDB）工程，采用 Cargo Workspace 组织多 crate。
整体目标是提供：
- 时序数据写入、查询与管理能力
- 分层的服务端架构（主控、数据节点、流式计算、元数据管理）
- 远程访问能力（当前以 gRPC 为主）
- CLI 与 C API 两类客户端接入方式
- 配套的逻辑测试、服务测试、磁盘性能测试与基准测试能力

Workspace 根配置显示核心成员包括：`os_base`、`db_utilities`、`db_common/commons`、`db_common/mdb`、`db_core`、`db_server`、`db_client`、`db_tools`、`tests/sqllogictest`。

## 2. 总体架构

### 2.1 分层结构

项目可以概括为以下几层：

1. 基础系统层
- `os_base`：对时间、线程、文件、系统信息、socket 等进行基础封装
- `db_utilities`：通用算法与工具组件（日志、配置、校验、容器结构等）

2. 公共能力层
- `db_common/commons`：SQL 词法/语法、逻辑计划、数据类型与消息定义、结果集表示等
- `db_common/mdb`：内置 KV/B+树相关存储组件（页、节点、缓存、树结构等）

3. 核心引擎层
- `db_core`：查询执行器、物理计划器、存储引擎（TSDB/WAL/TFS/Meta）、多角色服务引擎

4. 接入层
- `db_server`：服务端启动入口（`tzdb`）
- `db_client`：客户端 SDK（Rust + C ABI 导出）
- `db_tools`：交互式命令行工具（`tzdb-cli`）

5. 质量与验证层
- `tests/sqllogictest`：SQL 逻辑回归测试框架
- `tests/server_tests`：服务能力测试
- `tests/disk_test`：磁盘读写测试
- `benchmarks`：TPC-H 数据与查询基准（当前 workspace 中默认注释）

### 2.2 关键运行链路

典型链路（CLI 发起 SQL）如下：

`tzdb-cli(db_tools)` -> `db_client::TzdbContext` -> gRPC -> `db_server`/`db_core::msvr::transport` -> `db_core::query + storage`

执行过程中会经历：
- SQL 解析（lexer/parser）
- 语义转换（Translate）
- 逻辑优化（Analyzer + Optimizer）
- 物理计划生成（QueryPlanner）
- 执行计划下推到存储层（MetricScan/Filter/Aggregate/...）

## 3. 模块说明

### 3.1 `db_server`（服务启动器）

作用：
- 提供可执行程序 `tzdb`
- 调用 `MsvrEngine::parse_args()` 读取配置
- 启动主服务引擎 `MsvrEngine::run()`
- 拉起远程服务入口 `RemoteServer::run_server()`（当前实际主通道是 gRPC）

特点：
- 启动逻辑简洁，核心能力委托给 `db_core`

### 3.2 `db_core`（核心引擎）

`db_core` 是项目中最核心模块，包含三大域：`query`、`servers`、`storage`。

#### 3.2.1 查询域 `query`

模块：
- `catalog`：元数据读取与请求封装
- `query_execution`：Session/Context、DataFrame、物理计划构建入口
- `query_physical_expr`：表达式执行逻辑（列、字面量、函数、聚合）
- `query_physical_plan`：算子实现（Scan/Filter/Projection/Sort/Aggregate/Limit 等）
- `query_proto`：计划与 RPC 协议结构转换
- `scheduler`：调度相关（结构已预留）

策略：
- 采用“逻辑计划 -> 优化 -> 物理计划”分层模型
- `SessionState` 维护会话级信息（当前 DB、host/port、planner）
- `QueryPlanner` 负责将逻辑节点逐步转换为可执行物理算子树
- 聚合采用 `Partial + Final` 两阶段策略

#### 3.2.2 服务域 `servers`

核心角色：
- `msvr`：主服务引擎（管理与路由中枢）
- `cmngr`：集群/元数据管理能力
- `psvr`：数据相关引擎（含写入/查询/提交等）
- `csvr`：流式计算相关能力

核心机制：
- `NodeAction` + 函数指针表（消息类型 -> 对应处理函数）
- 多 runtime 模型：`QUERY_RUNTIME` / `RPC_RUNTIME` / `LOG_RUNTIME`
- 通过全局配置读取 `ServerIp`、`RpcPort`、`Role`、`MasterIp`、`MasterPort` 等
- 远程层支持扩展：模块中保留了 grpc/http/restful 结构，当前主实现为 grpc

#### 3.2.3 存储域 `storage`

模块：
- `tsdb`：时序存储主实现（memtable、文件集、读写器、压缩、校验）
- `wal`：预写日志
- `meta` / `meta_schema`：元数据结构
- `tfs`：文件存储抽象
- `oplog` / `elog`：日志相关结构

关键策略：
- 内存 + 磁盘结合：`Mutable`（内存）与文件集（磁盘）协同
- 时间范围建模：`TimeRange` 支持重叠、包含、交集等操作
- 写入持久化：WAL 记录日志并可按索引读取
- 数据组织：面向 TSKEY（时间戳键）与表 UID 管理
- 伴随元数据：通过 `mdb` 存储如 `TableDataInfo(min_key/max_key)` 等结构

### 3.3 `db_common/commons`（公共协议与 SQL 能力）

核心内容：
- SQL 词法：`query_lexer`
- SQL 语法：`query_parser`
- 语义翻译：`query_translate`
- 逻辑优化：`query_logical_optimizer`
- 查询表达式、结果集、数据块、类型系统、消息定义（`tmsg`）

策略：
- 通过 `Parser` 将 SQL 解析为 AST/Statement
- 通过 `Translate` 结合 catalog 元信息生成计划
- 在优化器中执行规则优化（filter/projection/limit 下推等）

### 3.4 `db_common/mdb`（内置索引/存储组件）

模块包含：`btree`、`pager`、`pagecache`、`db`、`tree` 等。

职责：
- 提供数据库内部所需的索引与页管理能力
- 与 `db_utilities` 协作实现持久化与缓存机制

文档 `doc/tdb/btree.md` 说明了 B+树的基本设计思想和删除流程。

### 3.5 `db_client`（客户端 SDK 与 C 接口）

能力：
- `TzdbContext::connect(host, port)` 建立 gRPC 连接并创建 session
- 提供数据库管理、表模型管理、写入、查询、消费等 RPC 调用封装
- 导出 C ABI（`connect.rs`、`capi.rs`、`statement.rs`）以便外部语言接入

现状：
- 核心查询/连接能力可用
- 多个 C API 接口仍为 `todo!()`，说明 C 层还在持续完善

### 3.6 `db_tools`（命令行工具）

可执行程序：`tzdb-cli`

能力：
- 连接远程服务（host/port）
- 交互式 REPL（基于 `rustyline`）
- 支持 SQL 执行与结果打印
- 支持 `\\q` 等命令

定位：
- 面向开发/测试阶段的运维与调试入口

### 3.7 `db_utilities`（通用工具库）

内容广泛，包括：
- 数据结构：skiplist、rbtree、queue、lrucache
- 编解码与校验：crc32、base64、md5
- 配置与日志：config、log4rs
- 错误定义：`tzerror`
- 宏与公共工具：macro_def、util、timer、uuid 等

策略：
- 用 feature 控制日志/配置组件组合（`yaml_format`、`toml_format`、`pattern_encoder` 等）
- 为核心模块提供可复用基础设施

### 3.8 `os_base`（系统抽象层）

提供跨平台基础封装：
- 时间/时区、线程、进程、目录、文件、socket、信号等

定位：
- 屏蔽部分平台差异，给上层模块提供统一调用接口

## 4. 配置与部署策略

### 4.1 配置加载

`MsvrEngine::parse_args()` 支持 `-c/--config` 参数加载配置文件。
关键配置项在示例 `tests/server_tests/src/cluster_test/tsdbconfig*/config.toml` 中可见，包括：
- `dataPath`
- `logPath`
- `ServerIp`
- `RpcPort`
- `Role`（Master/Slave）
- 从节点还可设置 `MasterIp`、`MasterPort`

### 4.2 角色策略

当前设计支持至少两种角色：
- Master：主节点
- Slave：从节点

在 gRPC 服务启动时，如果角色是 Slave，会执行连接到 Master 的逻辑（cluster connect）。

## 5. 查询与执行策略

### 5.1 SQL 处理流程

1. `Parser` 进行 SQL 词法+语法解析
2. `collect` 提取语句所需元数据
3. 通过 catalog 获取数据库/表 schema 信息
4. `Translate` 生成逻辑计划
5. `Analyzer + Optimizer` 做逻辑优化
6. `QueryPlanner` 生成物理计划
7. 执行算子树并返回结果集

### 5.2 计划优化策略

代码中已包含多种优化规则模块，例如：
- `push_down_filter`
- `push_down_projection`
- `push_down_limit`
- `eliminate_projection`
- `eliminate_filter`
- `simplify_expressions`

说明项目采取了基于规则的优化器策略。

## 6. 存储与可靠性策略

### 6.1 存储组织

- 时序主数据由 `tsdb` 模块维护
- 内存层使用 `Mutable` + 跳表（`SkipMap<TSKEY, TimeSeries>`）
- 磁盘层通过文件集与读写器组织
- 辅助信息可落到 `mdb`（如每表时间范围）

### 6.2 WAL 策略

`storage/wal` 提供：
- 日志追加写入
- 记录索引位置
- 可按编号或最后一条读取
- 支持 flush

该策略有助于写入路径上的持久化与恢复。

## 7. 流式计算与消息/订阅策略

`csvr`、`psvr::tq`、`cmngr` 等模块共同体现了流式与消息队列相关能力：
- 任务分发（fixed endpoint / shuffle）
- stream task 输入队列
- topic/consumer/subscribe/poll 等语义在 query plan 与 RPC 接口中均有体现

这说明项目不仅做静态查询，也在向“时序 + 流计算 + 订阅消费”一体化方向建设。

## 8. 测试与基准体系

### 8.1 SQL 逻辑测试

`tests/sqllogictest`：
- 基于 `sqllogictest-rs`
- 支持 `.slt` 用例
- 启动测试服务后逐文件执行

### 8.2 服务测试

`tests/server_tests` 包含：
- 创建/删除/写入/查询/多线程/监控/模型等测试代码
- `cluster_test` 提供主从配置样例

### 8.3 磁盘测试

`tests/disk_test` + `doc/disk_test/README.md`：
- 顺序读写、随机读写、混合读写
- 单线程与多线程版本
- 配套 fio 配置

### 8.4 基准测试

`benchmarks`：
- 提供 TPC-H 数据生成、转换、导入、查询流程说明
- 依赖 Arrow/Parquet/DataFusion
- 目前在 workspace 根成员里默认注释，可按需启用

## 9. 技术栈

### 9.1 语言与构建
- Rust 2021
- Cargo Workspace 多 crate 协作

### 9.2 核心库
- 异步与网络：`tokio`、`tonic`（gRPC）
- 序列化：`serde`、`serde_json`、`prost`
- 数据计算/格式：`flatbuffers`，并在 benchmarks 中使用 `arrow`/`parquet`/`datafusion`
- 并发与容器：`crossbeam-channel`、`dashmap`、`parking_lot`
- 命令行与交互：`clap`、`rustyline`

### 9.3 运行平台
- 代码中同时存在 Linux/Windows 相关适配（如 `winapi` 与 libc）

## 10. 当前工程特征与演进方向

从代码结构看，项目具备以下特征：
- 结构完整：具备从协议、执行、存储到工具链的完整闭环
- 架构可扩展：服务层预留 grpc/http/restful 多通道
- 场景复合：同时覆盖 TSDB、流式处理、订阅消费

同时也能看到部分模块仍在迭代：
- 若干接口保留 `todo!()`（尤其 C API 部分）
- workspace 中部分测试/基准成员默认未启用

整体上，这是一个偏“工程化内核 + 多能力融合”的 TSDB 项目，已经具备较完整框架与核心执行链路，适合继续在稳定性、协议完善和性能验证上持续推进。
