# RPC And Node Message Dispatch Flow

## 目标

为流式计算与节点执行提供统一远程调用通道，并保持与内部执行模型低耦合。

## 概念边界

- IPC:进程间通信
- RPC:远程过程调用

RPC 关注两件事:

1. 协议设计
2. 序列化/反序列化约定

## 处理链路(原文提炼)

在服务端同名函数中，处理流程为:

1. 接收 RPC 框架请求对象
2. 转换为 SQL 引擎请求对象(如 `MsgReq`)
3. 包装成内部消息(如 `RpcMsg`，含回传通道)
4. 投递到节点消息入口执行(如 `dm_process_node_message`)

## 框架选型记录

- `tonic`:作为主线实现路径
- `Volo`:做过调研与验证，后续记录为“已弃用方案”

原文对 Volo 的关注点:

- IDL 驱动代码生成机制
- 与既有网络模型(如 select)协同的不确定性
- 超大 IDL 的编译负担与优化空间
