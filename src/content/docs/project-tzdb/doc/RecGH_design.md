# RecGH 工程设计与项目意义

## 1. 项目定位

`RecGH` 是一个面向实时数据接入、分发、解析与入库的服务程序，核心目标是把“多源网络数据”统一转成可路由、可消费、可持久化的数据流。

从当前代码看，它主要面向以下场景：

- 外部网络与内部网络之间的数据桥接（TCP/UDP，含组播）
- 按消息内目标地址进行多目标转发
- 消息解析后写入数据库（通过 `libtzdb` / ODBC / 本地库接口）
- 航海相关数据协议处理（AIS、NMEA 等）

## 2. 总体架构设计

工程采用“发布-订阅-消费”模型组织数据流，主干结构清晰：

- `Publisher`：负责从网络连接读取原始包，产出 `PRODUCT_ELEM`（当前为 `FaiLInfo`）
- `Subscriber`：作为发布者与消费者之间的桥梁，维护消息队列与消费者组
- `Consumer`：对消息执行业务动作（转发、存储等）
- `SubscriberServer`：统一管理 IO、多线程任务分发与生命周期

对应实现主要位于：

- `inc/SubscribeModel.h` + `src/SubscribeModel.cpp`
- `inc/PublisherGroup.h` + `src/PublisherGroup.cpp`
- `inc/ConsumerGroup.h` + `src/ConsumerGroup.cpp`
- `inc/RecServerMain.h` + `src/RecServerMain.cpp`

## 3. 关键工程设计

### 3.1 统一消息载体（FaiLInfo）

`FaiLInfo` 既是传输协议体，也是系统内任务单元，封装了：

- 包头与载荷长度
- 目标主机列表（`HostList`）
- 处理策略（仅解析、仅发送、发送并解析）
- 数据类型与表名
- 业务载荷（`payload`）

该设计让“路由信息”和“业务数据”一起流动，减少了中间映射层。

### 3.2 网络层双角色支持

发布与消费端都支持 TCP/UDP，多处包含 server/client 两种行为，具备以下能力：

- TCP 监听并接入新连接（`create_new()`）
- UDP 单播与组播发收
- 在不同网卡/网段上绑定并发送

这使 `RecGH` 更像一个可编排的“网络数据中继器”而不是单一协议客户端。

### 3.3 异步并发模型

系统通过 `IOMode + RASThreadPool + DBSemaphore` 组合实现异步流水线：

- IO 线程负责等待可读事件并生成消息
- `Subscriber` 入队消息并通过信号量唤醒主任务流
- 工作线程池按“消息 x 消费者”展开任务执行

该模型把网络阻塞与业务处理解耦，便于扩展消费者类型。

### 3.4 可插拔解析与存储

在 `MainModule` 中，`DataAnalyze` / `DataStorage` 通过函数指针与表配置驱动：

- `pAnalyzeFuction`：按数据类型选择解析函数
- `pStorage` / `p_new` / 字典函数：按表结构选择入库策略

这套接口允许不同数据类型共用同一框架，仅替换“解析与落库插件”。

## 4. 数据流（端到端）

典型处理路径如下：

1. `Publisher` 从 socket 读取数据并反序列化为 `FaiLInfo`
2. `Publisher::dispatch` 将消息投递到关联 `Subscriber`
3. `Subscriber` 入队并唤醒任务线程
4. 每个 `Consumer` 根据策略执行：
5. `SendAllConsumer`：按 `HostList` 发送到多个目标地址
6. `StorageConsumer`：提取 payload 组装 SQL 并执行写库

在传统 `RecServer` 路径中，还会串行执行“接收-转发-解析-存储”并做运行状态统计。

## 5. 项目意义

`RecGH` 的价值不只在“转发”，而在于构建了一个领域可复用的数据中台雏形：

- 统一接入：屏蔽上游协议差异，提供一致数据处理入口
- 统一分发：把业务路由从网络代码中抽离到消息层
- 统一落库：让解析与存储按数据类型解耦扩展
- 统一监控基础：已有接收间隔、发送量、解析量、存储量等状态锚点

对航海/态势感知等强实时场景，这种结构可以显著降低新数据源接入和新消费链路上线成本。

## 6. 当前状态与演进方向

从代码可见，工程已经具备可运行骨架，但仍处于“可用 + 快速演进”阶段，建议下一步聚焦：

- 配置化完善：`tmpInit()` 仍带测试硬编码，配置文件驱动需要补全
- 内存与对象生命周期收敛：若干路径仍有手动内存管理风险
- 错误处理标准化：网络/DB 失败恢复策略需统一
- 领域模型收敛：`FaiLInfo` 可继续拆分“传输头”和“业务体”
- 可观测性升级：增加结构化日志与指标导出

总体上，`RecGH` 已具备“数据接入网关 + 分发总线 + 落库适配层”的核心雏形，对后续平台化建设具有直接意义。
