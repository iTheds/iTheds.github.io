## 背景

该记录整理自 `RemoteAccess模块系统设计` 原稿中 `9.28`、`9.30` 以及后续“日志锚点完善”部分。原稿的关键认识是:如果看不到系统关键步骤，连接池、状态保持、线程管理都会变成黑盒。

## 为什么需要监控体系

原稿直接指出:

- 现有系统没有办法看到关键步骤信息。
- 结构不透明会直接拉高维护难度。
- 监控不仅用于调试，也可以作为验证旧功能和辅助开发新功能的手段。

这不是把日志当作“补充输出”，而是把日志当成系统稳定性的基础设施。

## 计划监控的内容

原稿列出的监控项较完整，包含:

1. 被连接的数据库。
2. 每个数据库的连接数量，以及客户端 `ip`、`port`。
3. SQL 执行数量与分类统计，并记录最后时间。
4. 关键操作耗时，例如执行 SQL 的耗时。
5. 线程是否存活。
6. 允许其他组件复用监控体系记录自己的数据。

后续又进一步把职责拆到具体模块:

- `DBConnectPool`:被连接的数据库。
- `dbrConnect`:连接数量、客户端信息、SQL 分类统计、关键操作时间。
- `DBThreadManage`:线程存活信息。

## 挂载位置

原稿最终倾向把日志与监控系统挂在 `ENV` 顶层，但并不让 `ENV` 直接拥有所有细节，而是:

- 由 `ENV` 统筹与记录。
- 各主类下发信息收集模块。
- 通过其它模块提供必要数据。

这意味着 `ENV` 是汇总层，不是所有状态的原始拥有者。

## 输出策略

日志锚点部分已经形成较清晰的分级与落点。

### 信息分类

- 程序函数调用信息。
- 程序关键步骤执行情况。
- 程序分析内存数据信息，包括网络通信。

### 重要程度分级

- `LOG_LEVEL_ERROR`
- `LOG_LEVEL_WARNING`
- `LOG_LEVEL_DEBUG`
- `LOG_LEVEL_INFO`

### 输出位置

- `LOG_TARGET_CONSOLE`
- `LOG_TARGET_FILE`

### 约定策略

原稿给出的策略是:

1. 错误信息输出到文件。
2. 关键步骤输出到文件。
3. 步骤调试输出到控制台。
4. 输出到文件的信息也同步输出到控制台。

并约定优先使用:

- `LOG_IN_ERROR`
- `LOG_IN_INFO`
- `LOG_PRINTF`
- `LOG_DEBUG`

## 结论

这一记录的核心价值在于把日志从“printf 的集合”提升为了“系统可观测性”的一部分。对 RemoteAccess 这类连接、会话、线程、SQL 执行相互交织的模块来说，没有监控体系，后续的连接保持、重连、线程池与状态分析都很难稳住。
