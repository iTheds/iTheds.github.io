## 背景

该记录整理自原 `集群Rust项目Dev日志` 中 `12.28`、`12.29`、`1.9`、`1.10`、`1.19`、`1.23` 的连续设计思考，核心问题是:集群侧如何维护库表与节点之间的部署关系，并提供可扩展的检索方式。

## 元数据管理承担的职责

原始思路中，元数据并不只是“保存有哪些表”，而是承担整条数据通路的定位能力:

- 根据表名称找到目标地址，用于发送信息。
- 根据目标地址反查该节点持有哪些表。
- 在多副本、分区、副本一致或副本分区场景下，继续补充查询维度。
- 在执行查询、备份、同步时，依赖元数据判断目标分区或目标表位于哪个节点。

原始表述强调，查询条件和返回结果都必须面向后续扩展，而不是一次性写死:

> 通过 `库名:表名` 的格式进行检索，根据功能需求，每需要一个功能参数，就多一个查询条件。

## 从“单索引”到“多层结构”

早期方案直接把 `库名:表名` 作为关键字构造索引。这能解决表名重复问题，但马上暴露两个限制:

- 一个节点可能对应多个表，不是 `key -> node` 的一一映射。
- 一旦引入分区，索引键还要继续拼接 `partition id`。

因此记录中逐渐从“单一 map”转向“存储结构与索引结构分离”:

- 存储层只维护真实的部署信息。
- 检索层按查询需求建立额外索引。
- 添加/删除时，分别维护部署信息、存储信息和索引信息。

## 结构演化过程

原稿先后出现过两种结构草案。

第一种以节点为中心:

```text
meta manager{
    node set,
}
node {
    db set,
}
db{
    table set,
}
```

随后又意识到“节点中心”会导致部署结构过于臃肿，于是转向“部署信息单独维护，数据库存储结构独立存在”的思路:

```text
meta manager{
    node list,
    db set,
}
db{
    table set,
}
```

在这个方案里，真正可变的是“部署信息”，而不是数据库对象本身。最终原稿给出的落地规则是:

- 添加时:先校验或补齐存储信息，再添加部署信息，最后补建索引。
- 删除时:先删索引，再删部署信息，最后删存储信息。
- 查询时:优先通过索引命中，查询所有部署节点时再顺序检索。

## 查询返回方式

原稿没有把元数据层做成强策略组件，而是倾向于“返回原生结构，由上层自行处理”，包括:

- 对多结果集使用 `cursor + fetch` 的方式处理。
- 对上层暴露原生结构体结果。
- 在适当时机关闭和销毁查询游标。

这意味着元数据层更接近“事实存储 + 检索支持”，而不是“策略引擎”。

## 设计结论

这一阶段沉淀出的关键点有三条:

1. `库名:表名` 是必要的基础命名方式，但不能把它误当成唯一索引方案。
2. 存储结构与索引结构必须分离，否则分区、副本、多库场景会迅速把结构拖垮。
3. 元数据层主要负责提供定位能力，上层再决定调度、策略与结果处理。
