## 改造方向

| 方向 | 内容 |
| --- | --- |
| 程序形态 | 现有三个 `exe` 过于冗余，建议合并主程序与监控程序，或将监控程序单一化 |
| 界面形态 | 增加 QT 界面，使应用以软件而非控制台方式存在 |
| 网络体系 | 复用数据库 `server` 中的网络组件 |
| 数据库接口 | 更新为 `mco` 层接口 |
| 表结构 | 进一步细化表设计 |
| 可扩展性 | 提升自动执行能力与整体可扩展性 |
| 健壮性 | 通过总体设计、异常处理、线程隔离与数据控制保证稳定性，而不是再套一层“程序监控程序” |
| 运维能力 | 完善远程更新能力 |

补充细节:

- 增加每个类的输出方式。

## 详细设计

一个端口就可以支持对多个客户端的数据发送。所以船舶项目的端口数可以从 8*8 锐减到 1 。但是前提就是约定了一个通信协议，来指定发送何种数据。
次级选择就是，开启与数据数量同等量的端口，只要连接，就开始发送数据，所以只需要 8 个端口。
所以现在就分为两个部分，数据采集，数据订阅转发。

我们可以这样，设计一个只使用单端口进行通信的组件，该组件是可以复用的，直接复用到 远程访问服务 组件。
然后在本项目中，开辟若干个线程去响应。节省工作量。

程序过程：

读取采集信息表<标识了需要采集的数据项以及其 ip、地址等>，通过配置文件读取。

初始化网络：
分配相应量的线程<A, n>()进行收集数据。
分配监听连接的线程。

线程<A, n>()：
请求向上连接
建立连接
分配线程<B, 1>()进行专项接收数据。
该部分是否需要再开辟线程解析存储数据？如果开辟则需要进行线程间切换。
如果不需要则直接在该线程中完成此步骤。

线程<B, 1>()：
专门接收数据的线程。
是否有必要分配线程以接收数据呢。感觉必要性不是很大，因为socket本身就维护了一个缓冲区。

如果不需要<B, 1>的话，那么流程为：

1. 分配 n 个线程接收 n 个端口的数据。
2. 对数据进行存储。
3. 分配 n 个线程进行监听连接，生成相应数目的连接队列。
4. 根据线程中的连接队列，发送数据。

本次重构先把现有的功能整理，然后进行替换。

我们希望向上嗅探和向下分发两者分离开来。
之前的是通过结构体 SOCKET_INFO 将两者存放在一起。

设计判断如下:

1. 应先明确数据流和数据处理路径，再追加功能性模块。
2. 原程序通过 `recv` 判断连接断开并重连，更合理的做法是先完成当前数据发送与存储，再进入重连流程。
3. 适合通过配置表驱动子线程行为，减少分支散落在主流程中。
