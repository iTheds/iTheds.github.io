## 核心问题

原始日志里最重要的一条技术线，是如何把“协议解析”从固定长度读取升级为“协议驱动的分阶段读取”。其中心接口就是 `readSize`。

## readSize 的职责

原稿明确给出了接口语义:

- 返回值表示“下一次尝试读取的数据长度”。
- `read_buffer` 表示本次要读到哪个内存位置。
- `read_buffer_max_len` 表示缓冲区上限。
- `read_size` 初始必须为 `0`。
- `netRead` 通过反复调用 `readSize` 直到完整读取一个载荷包。

这个设计的关键不只是“能分段读”，而是把读流程的控制权交给协议对象，而不是交给连接层写死。

## TCP 方向的改造

原稿提出:

- 直接复用 `TcpProtocol` 中 `header_pb_` 与 `payload_pb_` 的内存。
- 去掉围绕旧 `payload_` 展开的打包思路。
- `pack` 函数直接根据入参构造可发送 `vector`。

这意味着 TCP 协议实现从“协议对象长期持有业务 payload”转向“按需组织发送数据”。

## UDP 分包的真实要求

原始记录反复强调，UDP 版本不能照抄 TCP:

- `pack` 不能把头和 payload 分开发送。
- 发送时必须按分包规则循环 `sendto`。
- 接收时也不能假设“一次收到完整载荷”，而应分阶段 `recvfrom`。
- 接收缓存应优先复用 `UdpProtocol::package_buffers_`。

原稿后来又加入了两个成员:

```cpp
int payload_len_ = 0;
int payload_max_len_ = 0;
```

目的就是在接收过程中复用 `payload_`，而不是每次重新分配。

## 顺序与组装问题

原稿没有回避 UDP 的现实限制:

- UDP 本身不能完全保证顺序。
- 因此可以考虑“边接收边组装到 `payload_` 中，全部完成后再解压”的方式。

这使得分包策略不仅是“把大包切小”，还包括:

- 包序信息如何携带。
- 接收阶段如何判定已完整。
- 缺包或乱序时是否允许重组。

## 为什么要做应用层分包

日志中的原始判断非常明确:

- UDP 合理传输范围 `< 64K`。
- 如果要支持更大内容，就必须进入应用层分包或流式传输。
- TCP 更适合大文件传输。

因此应用层分包不是“协议做得复杂一点”，而是为了摆脱底层 IP 分片不可控、不可观测的问题。
