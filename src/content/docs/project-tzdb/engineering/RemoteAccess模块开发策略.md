## 版本管理

原稿保留的 Git 仓库管理思路如下:

1. 维护 Windows 版本 GitLab，供局域网内访问。
2. 仅同步五个项目中的头文件与源码文件。
3. 收集常用指令。
4. 建立两个分支。

地址信息:

- `ip: 192.168.1.105`
- `http://localhost:3000/iTheds/tzdb-ras.git`
- `http://192.168.1.105:3000/iTheds/tzdb-ras.git`

原稿同时说明，待代码基本成型后，该临时仓库应废除，并转接到主仓库。

## 初期开发判断

原始判断集中在“先打通可运行 Demo，再逐段深入”:

- 在 `8.20` 前开发基本模型 Demo，使 Windows 驱动管理器至少可以识别并使用部分功能。
- 通信模块开发前，需要抓包或先查看 MySQL 的 `SQLExec()` 过程。
- 远程连接管理模块与连接池要同步考虑。
- `SQL_API` 开发必须建立在数据库知识和 TZDB 源码理解之上。
- 链路安全是不能绕开的单独问题。

## 讨论原则

原稿保留了一条非常直接的原则:

- 项目越存在技术盲点，调研成本越大，就越需要提供学习路径，让组内讨论其可行性。
- 特别在项目中期，如果对某件事高度纠结，优先遵循“第一想法原则”快速落地，再通过迭代修正。

## 结构重构考虑

原稿后续开始把重心放到代码结构清理上，主要关注:

- 重新整理命名、结构体和既有编码习惯。
- 明确以四个主要类为核心进行开发。
- 暂时不深入处理数据库源码中已有但当前阶段并不关键的那部分内容。
- 环境句柄、语句句柄、JDBC、Python ODBC 都还存在认知缺口，需要继续补课。

一个更现实的判断是:

- 当前最关键的并不是“把所有方向都铺开”，而是先把四个主类的职责分配讲清楚。

## 开发模型

原稿最终采用的是增量模型思路，并明确参考快速原型模型:

1. 先提出需求。
2. 再构建实现。
3. 再执行测试。
4. 重复这一过程，直到在满足已知模型的前提下确保交付。

开发期间强调的一条执行要求是:

- 先把整条路径实验通，确认无误后再正式开发。

## 测试与交付

测试方案:

- 新建项目 `tzdbodbc-test` 进行测试。
- 在测试过程中把 `tzdb-odbc` 编译为 `lib` 库，方便调试和观察实时内存数据。
