## 目标

该记录整理自原 `通信模块与IO模型开发日志（2025-05至2025-07）` 的 `2025-04-30` 到 `2025-05-05` 部分，保留原始设计重点，而不是把它压缩成泛泛结论。

## 时间线索

| 时间 | 关注点 |
| --- | --- |
| `2025-04-30` - `2025-05-01` | 先处理压缩、缓冲、字符串内存管理和工具层边界 |
| `2025-05-02` - `2025-05-03` | 引入 `ConnectCreator` 与生成器模式，重写连接构建过程 |
| `2025-05-04` - `2025-05-05` | 重新界定 TCP/UDP、协议对象、连接对象和 `net_pool` 的边界 |

当时的主线有四条:

- 重构当前通信模块，提升传输速率。
- 为 UDP 分包和后续大包传输留出协议空间。
- 用内存管理替换旧的字符串调整路径，减少重复申请释放。
- 梳理 TCP / UDP / 协议对象 / 连接对象之间的关系。

## 初始改造项

在 `2025-04-30` 到 `2025-05-01` 这一段，原始记录中的动作比较密集，主要包括:

- 引入 `lz4`、`zlib`，并预留后续压缩算法扩展能力。
- 增加用于优化发送顺序的优先级队列。
- 优化旧 KMP 算法，使其支持动态数据结构过滤。
- 去除 `AdjustString`，改用新的内存管理组件。
- 把 ODBC 相关接口从远程服务支持组件中移出到 `tools`。

这说明通信重构并不是只改协议头，而是连同压缩、缓冲、工具层边界一起重构。

## 生成器模式的引入

到 `2025-05-02` - `2025-05-03`，原稿明确指出，`CONNECT` 类违背了面向接口编程的思路，因为它把 TCP 和 UDP 的建连过程绑成了一套固定流程。针对这一点，提出了:

- 设计 `ConnectCreator`。
- 使用生成器模式描述不同连接的构建步骤。
- 由“指挥者”统一抽象不同生成器的生成顺序。

原稿保留了一个很关键的判断:

> 生成器模式是可以失败的，所以所有内容都放入到构造函数中不符合逻辑。

这句话决定了后续接口不会把“连接一定成功”当成默认前提，而是允许构建过程被显式管理。

## TCP / UDP 关系的重新界定

到 `2025-05-04` - `2025-05-05`，原稿重新梳理了通信对象之间的边界:

- TCP 和 UDP 都是点对点关系。
- UDP 可以进一步扩展到“点对集合”。
- TCP 通信组件不应长期持有数据体，最多只把数据放到 `net pool`。
- 协议对象与连接对象更接近依赖关系，而不是简单组合关系。

这也是后来把粘包、分包、缓存复用都收束到协议层处理的前提。

## 上层与底层的边界

同一阶段里还保留了几个没有被抹平的真实判断:

- `net_pool` 可以有缓存信道，但 `read/write` 更适合传入协议类型。
- TCP 的建立过程更像“任务”，可以有阻塞与非阻塞两种使用方式。
- 连接创建不应该只生成明确目标地址节点，而忽略本机地址节点。

这些表达说明，当时的设计重点不是“写一个统一接口”，而是防止接口统一之后把连接真实特性掩盖掉。
