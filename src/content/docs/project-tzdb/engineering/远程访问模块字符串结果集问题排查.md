## 问题现象

该记录整理自 `RemoteAccess模块系统设计` 原稿中 `12.04` 的连续排查内容。问题最初表现为:

- 客户端在 `new` 时提示堆已损坏。
- 问题稳定出现在 `testGetData` 路径。
- 更具体地说，指向 `tzdb_stmt_fetch` 相关流程。

## 排查路径

原稿通过多轮实验逐步收缩范围:

1. 只查询一次时，问题不出现。
2. 查询两次时，问题出现概率显著提升。
3. 同时存在空字符串与非空字符串时，更容易复现。
4. 只有空字符串，或者全部数据都为空时，问题并不出现。

原始记录最后给出了非常关键的判断:

> 发生问题的地方，并不是 select，而是 fetch。

因为 select 报错时，堆往往已经提前损坏。

## 定位结果

原稿最终把问题落到 `res_metaSingleStringTotStruct` 以及后续字符串更新路径上，并继续审查 `res_update`，发现:

- 更新字符串时存在“长度不足仍然拷贝”的情况。
- 旧字符串指针在复用前没有被正确释放。

修正动作是:

```cpp
RAS_DELETE(source_input.val.strValue);
```

修正后连续测试 20 次未复现，恢复旧实现后再次出现，说明定位基本成立。

## 结论

这份排查记录的价值在于:

- 它不是泛泛地说“字符串处理有 bug”，而是给出了复现条件。
- 明确了问题与“二次查询 + fetch + 字符串复用”相关。
- 最终把故障落到了可验证的释放缺失与长度处理问题上。
