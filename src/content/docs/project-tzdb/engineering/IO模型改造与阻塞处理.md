## I/O 模型的边界

原稿在 `2025-05-20` 明确提出一条原则:

> I/O 模型不承担连接管理，只负责单线程处理多连接事件。

据此，I/O 模型的职责被压缩为三类:

1. TCP listener 接收新连接。
2. TCP 接收数据。
3. TCP 处理关闭。

这一步很关键，因为它把“连接池”与“事件循环”拆开了。

## 主要问题

在 `2025-05-20` 后的测试里，围绕 `select` 与 `epoll`，原稿保留了几组非常具体的问题:

- `select` 阻塞时外部关闭 listener 会发生什么。
- 阻塞 `select` 期间手动 `close(socket)` 能否自然跳出。
- `epoll_wait` 阻塞时如何退出。
- 重复添加同一个句柄时，不同模型会出现什么行为。

原稿后续给出的结论是:

- `select` 关闭所有句柄描述符后可以自然跳出阻塞。
- `epoll` 不一定自动跳出。
- `select` 重复添加同句柄通常没有错误，因为本质上是位图。
- `epoll` 重复添加会返回 `EEXIST`。

## 代码整理方向

在同一轮改造里，工作并不是只加测试，日志里明确要求:

- 抽离公共接口到 `IOMode.cpp`，减少重复。
- 删除 `perror`，改为使用 `LOG_IN` 系列日志。

这说明当时已经意识到 I/O 模型代码存在结构重复和日志风格混乱的问题。

## 测试方式

原稿给出的测试方法不是单线程调用，而是明确按阻塞场景设计:

- 线程 A:启动 `iomode`、添加 listener、调用 `waitEvent`。
- 线程 B:创建 TCP 连接并发送数据。
- 线程 A:验证 `not_listen_pack_deal_in` 是否收到数据包。

这是因为 `waitEvent` 是阻塞函数，若用单线程测试，很多行为根本测不出来。

## 纯 UDP I/O 模型

到 `2025-06-10`，原稿进一步提出开发纯 UDP 的多路复用模型 `IoUdp`，目标是:

- 不依赖 `select/epoll`。
- 用 UDP 连接替代事件句柄。
- 初始化时默认绑定本地 UDP 端口。
- `waitEvent` 直接阻塞在 `recv` 上。

这不是传统意义上的“多路复用器”，更像是为全 UDP 场景单独封装的一层阻塞式 I/O 模型。
