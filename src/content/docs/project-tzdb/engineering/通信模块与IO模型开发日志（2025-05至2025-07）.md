## 背景与目标

该记录聚焦嵌入式数据库项目中的通信协议与网络组件工作，主目标包括：

- 重构通信模块，提升传输速率。
- 规划并实现 UDP 分包机制（需求：通信包 `< 32K`）。
- 研究不依赖系统事件的 UDP I/O 多路模型。
- 引入内存管理，降低 TCP 粘包处理中反复申请释放开销。
- 补齐测试体系，并完成 613 文档整理。

初始问题报告：

- 单条查询耗时：Windows 上 `UDP ≈ 5000 us`，`TCP ≈ 500 us`。
- Linux 上 `UDP ≈ 50 us`，`TCP ≈ 67 us`。
- 初步诊断：不同系统 I/O 模型表现与配置差异导致。

### 2025-05-04

- 协议类需重点处理多线程读写冲突；无法内聚时应上抛结构到上层。
- `net_pool` 可有缓存信道，但 `read/write` 更适合传入协议类型。
- 协议与连接关系更偏依赖而非组合。
- 设计 `ProtocolFormat::readSize` 方案：
  - 返回下次读取长度；
  - `read_buffer` 指示下次读取目标指针；
  - `read_buffer_max_len` 表示缓冲上限；
  - `read_size` 首次必须为 0；
  - `netRead` 反复调用 `readSize`，直到完整读取载荷包。
- 提出直接复用 `TcpProtocol` 中 `header_pb_` 与 `payload_pb_` 内存，减少重复分配。
- 讨论 UDP 版本实现：支持分包，允许扩展 `UdpProtocol` 成员函数与成员变量。
- 要求修改 `TcpProtocol` 的 `pack`：去掉围绕 `payload_` 的旧模式，直接基于入参构造可发送 `vector`。
- 强调 UDP 发送应循环 `sendto` 分包，接收应分阶段 `recvfrom`，不应“先拼大包再发”。

- TCP 连接创建从“任务式阻塞建连”与“上层显式使用”两种思路反复权衡。

协议实现反思：

- `UdpProtocol::pack` 将头和 payload 分开发送不可行，应按分包规则组织发送单元。
- `readSize` 应聚焦“下一次读多少、读到哪里”，并服务于 `netRead` 的循环读取模型。
- 建议接收缓存优先复用 `UdpProtocol::package_buffers_`。
- 考虑 UDP 无序特性：边接收边组装到 `payload_`，完成后再解压。
- 新增 `UdpProtocol` 复用字段：
  - `int payload_len_ = 0`
  - `int payload_max_len_ = 0`
- 将 `readSize` 拆分为多个函数以优化流程。
- 记录“应用层分包必要性”判断：
  - UDP 合理传输范围 `< 64K`，可做流式大文件传输；
  - TCP 更适合大文件传输。

补充知识点：

- 组播（多播）/广播/单播区别。

### 2025-05-20

- 开始改造 I/O 模型，目标是完成后接入数据库联调。
- 原则：I/O 模型不承担连接管理，只负责单线程处理多连接事件。
- I/O 模型职责定义：
  1. TCP listener 接收新连接
  2. TCP 接收数据
  3. TCP 处理关闭
- 研究问题：
  - `select` 中 listener 如何优雅关闭；
  - 阻塞 `select` 期间外部 `close` 的行为；
  - 阻塞场景如何跳出等待；
  - `epoll_wait` 阻塞退出方案。
- 代码规整方向：
  - 抽离公共接口到 `IOMode.cpp` 降低重复；
  - 删除 `perror`，改用 `LOG_IN` 系列日志。
- 测试方法调整：
  - `waitEvent` 是阻塞函数，测试应采用多线程；
  - 线程 A：启动 `iomode` + listener + `waitEvent`；
  - 线程 B：建立 `tcp con`、连接成功后发送数据；
  - 线程 A 验证 `not_listen_pack_deal_in` 收包。
- 模型差异记录：
  - `select`：关闭所有句柄描述符可跳出阻塞；
  - `epoll`：不一定自动跳出；
  - `select` 重复添加同句柄（位图）通常无错误；
  - `epoll` 重复添加返回 `EEXIST`。

- 后续规划：

- `NetPool` 简化方向：
  1. 去除多级多类任务队列
  2. 优化图谱化连接管理
  3. 任务线程以 I/O 线程为主，支持多线程任务
- 深层设计问题：
  - 连接类型与协议类型均可变；
  - 同一 `NetPool` 持有多连接但统一视为 `CONNECT`；
  - 发送/接收阶段要求“一种连接对应唯一协议”；
  - 协议工具既要解析加工，也要可复用以降低分配开销；
  - 核心矛盾在“连接管理、协议复用、协作关系”三者平衡。

## 性能测试记录

## 延迟测试（摘要）

- 100 次测试：平均 `150.32 us`，标准差 `106.464 us`
- 增加缓冲后：平均 `103.405 us`，标准差 `22.4964 us`
- 300 次测试：平均 `117.365 us`，标准差 `61.1449 us`
- 日志耗时影响后：平均 `115.993 us` / `142.62 us`（不同轮次）
- 无日志情况下可到 `89.23 us`，有日志时可上升到 `158.968 us`

## 发送与接收性能（摘要）

发送测试（1000 包）：

- `1024 bytes`：平均 `7.955 us`
- `4096 bytes`：平均 `15.846 us`
- `16384 bytes`：平均 `44.109 us`
- `65536 bytes`：平均 `129.577 us`

接收测试（忽略首次等待后，98 包）：

- `1024 bytes`：平均 `3.91837 us`
- `4096 bytes`：平均 `14.9694 us`
- `16384 bytes`：平均 `46.8878 us`
- `65536 bytes`：平均 `168.796 us`

错误记录：

- 出现 `netRead errno:9`（文件描述符无效）日志，需与连接关闭路径联动排查。

## 端到端性能（摘要）

`EndToEndPerformanceTest` 记录：

- `1024 bytes`：总均值 `8.388 us`，丢包率 `0%`
- `4096 bytes`：总均值 `24.2993 us`，丢包率 `0.1%`
- `16384 bytes`：总均值 `90.4094 us`，丢包率 `0.1%`
- `65536 bytes`：总均值 `317.397 us`，丢包率 `0.1%`

分析备注：

- `sendAndReceive` 统计逻辑可能引入额外开销；
- 阻塞 `recv` 与发送间隔耦合，影响可测性；
- 长消息分包对性能损耗明显。
