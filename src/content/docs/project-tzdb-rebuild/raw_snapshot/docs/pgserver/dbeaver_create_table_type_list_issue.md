# DBeaver Create Table 类型列表问题说明

## 问题现象

在 DBeaver 连接 `pg_server` 后，打开 `Create New Table` 的列类型下拉框时，曾经出现以下问题:

- 只显示 `varchar`
- 显示重复类型
- 显示数据库实际不支持的类型，例如 `text`、`numeric`、`bytea`
- 选择部分类型后，真正执行 `CREATE TABLE` 会失败

对比 PostgreSQL 本身时，这个现象会更明显，因为同一个 DBeaver 在连接 PostgreSQL 时能够正常显示完整且可用的类型集合。

## 根因

根因不在 DBeaver 写死的静态列表，而在 `pg_server` 返回的 PostgreSQL 系统目录元数据。

DBeaver 的 PostgreSQL 插件在建表类型下拉场景下，会读取 `pg_catalog.pg_type`、`pg_namespace` 等系统目录，并按 schema 缓存类型。也就是说，UI 里看到的类型列表，本质上是 `pg_server` 对 `pg_type` 的兼容结果。

这次问题的根因分成两层:

### 1. `pg_type` 初期兼容不完整

早期 `pg_server` 在 DBeaver 需要的几条关键元数据链路上与 PostgreSQL 不一致，例如:

- prepared statement 参数类型 OID 没有正确保存和回传
- synthetic `pg_type` / `pg_namespace` 某些字段把 SQL `NULL` 发成了空串
- `regproc` 相关列没有按 PostgreSQL 风格返回

这些问题会导致 DBeaver 的 type cache 构建不完整，最开始甚至退化成只剩 `varchar`。

### 2. “协议兼容类型”和“真实可建表类型”混在了一起

在后续补齐 `pg_type` 兼容后，DBeaver 已经能正常读取类型缓存，但又暴露出第二个问题:

- `pg_server` 的 synthetic `pg_type` 曾把 `numeric`、`bytea` 这类类型标成用户可见
- 但 TZDB 当前 `CREATE TABLE` 绑定逻辑并不支持这些类型

换句话说，协议层此前维护的是一份“为了 PostgreSQL/JDBC 元数据兼容而存在的类型集”，而不是一份“数据库真实支持的建表类型集”。

这会产生一个典型错配:

1. DBeaver 从 `pg_type` 里看到某个类型
2. 用户在建表向导里可以选到它
3. 真正执行 `CREATE TABLE` 时，TZDB binder 报 `unsupported type`

## 为什么不能直接把这些类型从 `pg_type` 全删掉

因为 DBeaver/JDBC 不只是把 `pg_type` 用作建表下拉，还会在其他元数据路径里按 OID 查询系统类型。

例如下面这些类型，即使当前不应该出现在建表下拉里，也仍然可能被客户端按 OID 查询:

- `oid`
- `text`
- `bytea`
- `numeric`

如果把它们从协议兼容层彻底删除，DBeaver 的其他元数据查询可能再次失败。

因此，这里必须把两类语义拆开:

- 建表可选类型:只保留 TZDB 当前真实支持的类型
- 系统兼容类型:继续允许按 OID 解析，以满足 JDBC/DBeaver 的元数据访问

## 修复方案

修复后的策略是:

1. `pg_server` 在类型缓存/列表查询路径中，只暴露当前真实支持的建表类型
2. `oid`、`text`、`bytea`、`numeric` 等系统兼容类型，不再进入 DBeaver 的建表类型缓存
3. 这些类型仍然保留按 OID 查询的兼容能力，避免 JDBC 元数据路径回退

当前 `CREATE TABLE` 实测支持和不支持的结论如下:

- 支持:`bool`、`int2`、`int4`、`int8`、`float4`、`float8`、`date`、`time`、`timestamp`、`varchar`、`smallserial`、`serial`、`bigserial`
- 不支持:`text`、`numeric`、`decimal`、`bytea`

其中:

- `decimal` 在 parser/binder 路径里会落到 `numeric` 语义，因此当前同样不支持
- `smallserial` / `serial` / `bigserial` 作为 PostgreSQL 风格语法糖，当前会在 binder 中翻译成对应整型列，并附加 `GENERATED BY DEFAULT AS IDENTITY` 语义与 `NOT NULL`
- 这三个名字不是 PostgreSQL `pg_type` 里的真实基础类型，因此协议兼容层仍不需要把它们伪造成 `pg_type` 行；即使 DBeaver 客户端自行展示这些别名，服务端现在也能正确执行 `CREATE TABLE`

## 代码定位

当前实现里，建表可见类型和元数据兼容类型已经拆开:

- `src/pgserver/pg_protocol.cpp`
  - `SupportedPgTypeDef::create_table_supported`
  - `ForEachSupportedPgType(...)`
- `src/binder/bind_create.cpp`
  - `Binder::BindColumnDefinition(...)`

前者决定 DBeaver 在 `pg_type` cache 里能看到什么，后者决定 TZDB 真正能否执行 `CREATE TABLE`。

## 后续约束

后续只要新增或删除 `CREATE TABLE` 支持的类型，就必须同时检查两处:

1. binder 是否真正支持该类型
2. `pg_server` 是否应该把该类型暴露到 DBeaver 的建表类型列表

原则是:

- 不能让 UI 显示数据库不支持的建表类型
- 也不能为了隐藏 UI 类型，破坏 JDBC 对系统类型的元数据访问
