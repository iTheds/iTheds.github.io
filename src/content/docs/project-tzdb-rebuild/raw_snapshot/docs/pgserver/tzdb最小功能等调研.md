# TZDB 最小功能调研报告

## 需求调研

主要需求包括：

1. **最小功能集裁剪**：通过界面工具选择功能集合，生成可直接使用的产物；
2. **直接读取文件方式的界面工具**：考虑 shell 的可裁剪性；
3. **直接读取内存数据库的工具**。

## 设计方案

### 基础工具选择

基于现有工具(如 DBeaver 和 SQLiteStudio)进行开发，主要需要实现：

1. 直接读取文件的接口；
2. 编译最小化裁剪工具的页面。

### 工具对比

- **DBeaver**：支持 C/S 架构，可在运行时访问内存数据库；
- **SQLiteStudio**：面向文件，但需要额外开发 C/S 架构支持。

### 应用跟随运行方案

两种实现方案：

1. **纯界面形式**：通过命令行实现最小化功能集和应用运行。若需状态监控，则采用 C/S 结构；
2. **代码嵌入方式**：工具运行时启动进程跟随，好处是无需通信即可监控数据库状态。

## 最小化功能裁剪界面

### 目标

通过选择不同库，实现功能选择。在不提供源码和环境的条件下，支持本机生成和使用。

### 实现方式

数据库提供多个功能集的 obj 文件，用户选择并通过逻辑限定范围和依赖条件，生成总的 lib 文件供应用使用。最小功能集不改变对外接口，确保编译阶段接口可用。

## 工具可行性比较

从语言、接口、实现难度等方面比较。界面部分均为可执行文件。

## SQLiteStudio

- **技术栈**：使用 Qt 编写，可通过 QtCreator 运行；
- **环境问题**：Qt 环境配置存在问题，Copilot 未使用；
- **接口实现**：可直接实现其接口，继承 AbstractDb 接口；
- **局限性**：面向文件，无 C/S 架构，若需通信协议需重新搭建。

## DBeaver

- **优势**：全部通过协议确定，界面纯粹执行命令行；
- **状态监控**：需 JNI 接口或实现 C/S，涉及 Java 代码编写；
- **接口需求**：需实现 PostgreSQL 接口。

## 总结与下一步

- DBeaver 更适合 C/S 架构和内存数据库访问；
- SQLiteStudio 适合文件读取，但扩展性有限；
- 建议优先实现 DBeaver 的 PostgreSQL 协议接口，并开发最小化裁剪界面。
