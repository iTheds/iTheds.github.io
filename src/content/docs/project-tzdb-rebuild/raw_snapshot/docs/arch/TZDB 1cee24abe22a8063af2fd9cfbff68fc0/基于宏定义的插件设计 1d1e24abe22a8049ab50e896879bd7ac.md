# 基于宏定义的插件设计

插件设计可以从以下两个方面考虑：

- 面向程序开发者(能够编译系统的人员)
- 面向用户

面向开发者可以通过在编译内核时直接定义相关的宏，以支持不同的功能；而面向用户应该考虑当用户链接一个目标静态库时需要哪些操作才能够将功能加载进来。综合以上考虑，进行如下设计以支持插件功能：

### 将功能进行抽象

这里以存储为例， 在内核代码中将功能抽象成一系列宏定义，并且在调用时只调用这些抽象定义：

```cpp
// 在内核源文件中定义.
extern struct tzdb_disk_storage_vtable_t_ disk_storage_vtable;

#ifdef TZDB_CFG_DISK_STORAGE_SUPPORT
	#define tzdb_disk_storage_init_wrapper(params)\
		(disk_storage_vtable.init ? disk_storage_vtable(params) : 0)
	/* 其他定义 */
#else
	#define tzdb_disk_storage_init_wrapper(params) 0
#endif

typedef struct tzdb_disk_storage_vtable_t_
{
	int (*init)(const char* cfg);
	/* 其他定义 */
}
```

定义后，在内核需要的位置调用`tzdb_disk_storage_init_wrapper` 来完成调用指定的函数，而当`disk_storage_vtable`没有被初始化时，则是一个无效调用。

### 定义用户接口

在用户层文件中(也可以时提供给用户的插件专属头文件如`disk_storage.h`)添加一个插件初始化函数，来初始化存储插件:

```cpp
EDB_RET tzdb_disk_storage_init();
```

这个函数的实现在`disk_storage.a`中， 这是一个静态库连接的桥梁，当用户没有链接`disk_storage.a`时，将编译不通过，同样的，若链接`disk_storage.a`却不调用这个初始化函数，那么编译器将抛弃`disk_storage.a`的相关符号。

### 实现用户接口

disk_storage.a的实现:

```cpp
extern struct tzdb_disk_storage_vtable_t_ disk_storage_vtable;

int init(const char* cfg) {
	// ...
	return 0;
}

EDB_RET tzdb_disk_storage_init() {
	disk_storage_vtable.init = init;
	
	return EDB_OK;
}
```

### 编译(开发人员)

```cpp
cmake .. -DDISK_SUPPORT=1
```

通过这次编译，将生成两个库`libkernel.a`(支持磁盘存储的)、`libdisk_storage.a`

### 编译(用户)

```cpp
gcc main.cpp -o a.out -l kernel disk_storage
```

### 用户层代码

```cpp
int main() {
	EDB_RET rc = EDB_OK;

	rc = tzdb_disk_storage_init();

	/* 其他数据库代码 */
	
	return 0;
}
```

通过以上内容就完成了基于宏定义的插件设计。
