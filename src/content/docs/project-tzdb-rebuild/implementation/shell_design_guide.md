# 管理工具 Shell

TZDB Shell 是一个交互式命令行数据库管理工具，为用户提供直接的数据库操作接口。该工具通过 SQL 命令执行数据查询、修改和管理操作，支持完整的事务生命周期管理和实时状态反馈。

TZDB Shell 的设计目标在于提供一个高效的数据库交互平台，支持快速的开发迭代、测试验证和运维管理。相比于编写完整的应用程序代码，通过 Shell 可以直接执行 SQL 命令并获得即时反馈，显著提高了开发效率。该工具适用于以下场景：数据库功能验证、SQL 查询调试、临时数据操作、数据库结构管理和问题诊断。

TZDB Shell 将复杂的数据库操作抽象为简单的命令行接口。用户仅需掌握基本的 SQL 语法即可完成大部分操作，而底层的连接管理、事务协调、资源分配等细节由系统自动处理。这种设计遵循关注点分离原则，使用户能够专注于业务逻辑而无需关注实现细节。


## 设计目标与动机
### 接口简洁性

### 交互式执行模型

TZDB Shell 采用 REPL（Read-Eval-Print Loop）交互模型，支持命令的即时执行和结果反馈。用户输入命令后，系统立即执行并返回结果，形成快速的反馈循环。这种模型特别适合于探索性查询、增量式开发和问题诊断。

### 事务一致性保证

TZDB Shell 实现了完整的 ACID 事务语义。所有数据修改操作要么全部提交，要么全部回滚，确保数据库始终处于一致状态。这种强一致性保证是数据库系统的基本要求，也是 TZDB Shell 的核心设计约束。

## 管理功能

### 用户界面

TZDB Shell 提供一个基于命令行的交互式用户界面。系统采用 REPL（Read-Eval-Print Loop）模型，用户在命令行提示符处输入命令，系统立即执行并返回结果。提示符动态显示当前的执行上下文：默认提示符为 `tzdb>`，表示无活跃事务；当启动事务后，提示符变为 `txn <id>(<state))>`，显示事务 ID 和状态；多行输入时显示续行提示符 `...`。系统使用 libfort 库将查询结果格式化为美观的表格，包含列标题、数据行和边框，支持 UTF-8 编码。

### 支持的指令体系

TZDB Shell 支持三类指令：SQL 查询指令、事务控制指令和系统控制指令。SQL 查询指令包括完整的 DDL（CREATE、ALTER、DROP）、DML（INSERT、UPDATE、DELETE）和 DQL（SELECT）操作。事务控制指令包括 BEGIN/\BEGIN（启动事务）、COMMIT/\COMMIT（提交事务）、ROLLBACK/\ROLLBACK/ABORT/\ABORT（回滚事务）。系统控制指令包括 \q、quit、exit（退出程序）和 --help（显示帮助）。

### 启动与连接过程

TZDB Shell 的使用从命令行启动开始。用户可以通过多种方式指定数据库位置。最简单的方式是直接运行 `shell` 命令，此时系统使用默认数据库 `./test_shell`。如果需要指定特定的数据库，用户可以提供数据库名称，例如 `shell mydb` 将连接到 `./mydb`。对于存储在特定路径的数据库，用户可以同时指定路径和名称，例如 `shell /data/db mydb` 将连接到 `/data/db/mydb`。系统支持的命令行选项包括 `--emoji-prompt`（使用 emoji 提示符）、`--disable-tty`（禁用 TTY 模式用于脚本输入）、`--no-txn`（禁用自动事务模式）、`--help`（显示帮助信息）。

启动后，TZDB Shell 初始化数据库连接并加载内部组件（事务管理器、数据目录、SQL 上下文）。系统初始化命令行编辑库（linenoise-ng）以支持历史记录和行编辑功能。系统显示欢迎信息并进入交互循环。此时命令行显示默认提示符 `tzdb>`，表示系统已准备好接收用户命令。

### 基本查询执行

用户可以在提示符处输入 SQL 查询命令。对于简单的单行查询，用户直接输入 SQL 语句并以分号结尾，例如 `SELECT * FROM users;`。系统接收到分号后立即执行查询，将结果格式化为表格形式输出。表格包含列标题、数据行和边框，便于用户阅读。

对于复杂的多行查询，用户可以跨越多行输入。系统在接收到第一行后，如果该行不以分号结尾，则显示续行提示符 `...` 并等待下一行输入。用户可以继续输入查询的后续部分，直到输入以分号结尾的行。系统在接收到完整的查询后执行。这种方式允许用户格式化复杂的 SQL 语句，提高可读性。例如：

```
tzdb> SELECT id, name, email
... FROM users
... WHERE age > 18
... ORDER BY name;
```

系统在接收到最后一行的分号后执行整个查询，将结果以表格形式输出。

### 事务管理过程

TZDB Shell 支持显式的事务管理。用户通过 `BEGIN` 或 `\BEGIN` 命令启动一个新事务。系统创建事务对象并返回事务 ID 和初始状态。此后，命令行提示符变为 `txn <id>(RUNNING)>`，表示当前处于活跃事务中。

在事务中，用户可以执行多个 SQL 操作（INSERT、UPDATE、DELETE、SELECT）。每个操作在事务的上下文中执行，但不立即提交到数据库。如果操作执行失败，系统捕获异常并将事务标记为 `TAINTED` 状态，提示符变为 `txn <id>(TAINTED)>`。此时用户必须显式执行 `ROLLBACK` 命令来撤销事务中的所有修改。

如果事务中的所有操作都成功执行，用户可以执行 `COMMIT` 或 `\COMMIT` 命令来提交事务。系统将事务中的所有修改永久保存到数据库，事务状态变为 `COMMITTED`，提示符恢复为默认的 `tzdb>`。

如果用户在任何时刻决定放弃事务中的修改，可以执行 `ROLLBACK` 或 `\ROLLBACK` 命令。系统撤销事务中的所有修改，事务状态变为 `ABORTED`，提示符恢复为默认的 `tzdb>`。

### 自动事务模式

在默认配置下，TZDB Shell 运行在自动事务模式。用户无需显式执行 BEGIN 和 COMMIT 命令。每个 SQL 查询自动被包装在一个事务中：系统启动一个隐式事务，执行用户的 SQL 命令，然后自动提交事务。如果执行失败，系统自动回滚该事务。这种模式适合于单个查询的快速执行，用户无需关注事务管理的细节。

### 错误处理与恢复

当 SQL 查询执行失败时，系统捕获异常并输出错误信息到标准错误流。错误信息包含失败的原因，帮助用户理解问题所在。在自动事务模式下，失败的查询自动回滚，不影响数据库状态。

在显式事务模式下，如果事务中的操作失败，系统将事务标记为 TAINTED 状态。此时用户必须显式执行 ROLLBACK 命令来清除污染状态。这种设计强制用户显式处理错误，避免在错误状态下继续执行操作导致数据不一致。用户可以根据错误信息修正 SQL 语句，然后重新执行。

### 结果查看与分析

TZDB Shell 将查询结果格式化为表格形式输出。表格包含列标题行和数据行。列标题清晰显示每列的名称，数据行显示查询返回的记录。表格使用 UTF-8 编码，支持中文等多字节字符。系统自动处理列宽度和对齐，使表格易于阅读。

对于返回大量数据的查询，系统逐行输出表格，用户可以在终端中滚动查看。系统支持简化输出模式（通过 `BeginTable(true)` 调用），此时表格不显示边框，仅显示数据，便于脚本处理。

### 命令历史与编辑

TZDB Shell 使用 linenoise-ng 库提供命令行编辑功能。系统维护最多 1024 条命令历史记录。用户可以使用上下箭头键浏览历史命令，快速重新执行之前的命令。系统支持行编辑功能，用户可以使用左右箭头键移动光标，使用 Backspace 删除字符，使用 Delete 删除光标后的字符。这些功能大大提高了交互效率。

### 退出与清理

用户可以通过多种方式退出 TZDB Shell。最常见的方式是输入 `\q`、`quit` 或 `exit` 命令。系统接收到退出命令后，首先检查是否存在未提交的活跃事务。如果存在，系统自动回滚该事务，防止数据不一致。然后系统释放命令行历史记录等资源，关闭数据库连接，退出程序。

用户也可以通过按下 Ctrl+D（在 Unix/Linux 系统中）或 Ctrl+Z（在 Windows 系统中）发送 EOF 信号来退出。系统同样执行清理操作。即使用户意外退出（例如强制关闭终端），TZDB Shell 的清理机制也确保了数据库的一致性。

### 脚本模式与自动化

TZDB Shell 支持非 TTY 模式（通过 `--disable-tty` 选项启用），允许从脚本或管道输入命令。在此模式下，系统从标准输入读取命令，无需 linenoise 库提供的交互功能。这使得 TZDB Shell 可以集成到自动化脚本中，用于批量执行数据库操作。

例如，用户可以创建一个包含 SQL 命令的文本文件，然后通过管道将其输入到 TZDB Shell：`cat commands.sql | shell --disable-tty mydb`。系统逐行读取命令，执行相应的操作，输出结果。这种方式适合于需要重复执行的数据库操作。




## 特殊设计特性

### 事务污染机制

TZDB Shell 实现了事务污染（TAINTED）状态机制以增强错误安全性。当事务中的操作执行失败时，事务被标记为 TAINTED 状态。此状态的目的是防止用户在错误状态下继续执行后续操作，从而导致数据不一致。用户必须显式执行 ROLLBACK 命令才能清除污染状态。这种设计强制用户在错误发生时进行显式的错误处理，避免了隐式的错误恢复可能带来的数据不一致问题。

### 双模式事务执行

TZDB Shell 支持两种事务执行模式。自动事务模式（默认）下，每个 SQL 查询自动被包装在一个事务中，执行完成后自动提交，适合于单个查询的快速执行。显式事务模式（通过 --no-txn 标志禁用自动模式）下，用户通过 BEGIN、COMMIT、ROLLBACK 命令显式管理事务边界，适合于需要原子性执行多个操作的场景。这种设计在简易性和灵活性之间取得平衡。

### 多行查询输入处理

TZDB Shell 支持跨越多行的查询输入。系统通过检测行尾的分号或反斜杠来判断查询是否完成。在 TTY 模式下，系统使用 linenoise 库提供的行编辑功能；在非 TTY 模式下，系统通过标准输入读取。这种设计允许用户输入格式化的多行 SQL 语句，提高了可读性和可维护性。


## 应用场景

### 开发与测试

TZDB Shell 为软件开发和测试提供了高效的工具支持。开发人员可以通过 Shell 快速验证 SQL 查询的正确性，无需编写完整的应用程序代码。测试人员可以使用 Shell 进行测试数据的设置、数据库行为的验证、以及数据一致性的检查。这种快速反馈循环显著提高了开发效率。

### 数据库管理与维护

数据库管理员可以通过 TZDB Shell 执行 DDL 操作（表创建、修改、删除）、索引管理、以及各类维护操作。Shell 提供了直接的命令行接口，避免了复杂的图形界面或脚本编写的开销。

### 数据探索与分析

TZDB Shell 支持交互式的数据查询和临时分析。用户可以快速浏览数据库结构、执行探索性查询、进行数据统计分析。这对于数据分析师和业务人员进行临时查询特别有用。



## 架构与实现

### 核心组件架构

TZDB Shell 的实现基于分层架构设计。最上层是用户交互层，包括命令行输入处理（ReadQuery 函数）、命令识别和分类（IsExitCommand、IsTransactionControlCommand）、提示符管理（GetTransactionPrompt）。中间层是业务逻辑层，包括查询规范化（NormalizeQuery）、事务管理（HandleTransactionalCmd）、SQL 执行（SqlContext）。最下层是数据库引擎层，包括数据库连接（Database）、事务管理器（TransactionManager）、数据目录（Catalog）。

### 关键类与接口

FortTableWriter 类实现了 ResultWriter 接口，提供了统一的结果格式化机制。该类通过 fort::utf8_table 对象维护表格状态，支持表头、数据行、边框的动态构建。WriteCell 和 WriteHeaderCell 方法负责单元格数据的写入，BeginTable 和 EndTable 方法管理表格的生命周期。

系统通过 SqlContext 与数据库核心引擎交互，隔离了 Shell 与数据库引擎的耦合。SqlContext 提供了两种执行方式：Execute 方法用于在显式事务中执行查询，PollExecuteGeneralSql 方法用于自动事务模式执行。

事务管理通过 Transaction 对象实现，支持完整的事务生命周期。Transaction 对象维护事务 ID、状态、读时间戳、提交时间戳等信息。系统通过 GetTransactionState、SetTainted、GetReadTs、GetCommitTs 等方法提供事务状态查询接口。

### 命令处理流程

TZDB Shell 的命令处理遵循以下流程：首先通过 ReadQuery 函数读取用户输入，支持多行输入和 TTY/非 TTY 两种模式；然后通过 NormalizeQuery 函数规范化查询字符串，移除尾部空格和分号；接着通过命令识别函数（IsExitCommand、IsTransactionControlCommand）判断命令类型；最后根据命令类型调用相应的处理函数。

对于 SQL 查询，系统根据当前是否存在活跃事务选择执行模式。如果存在活跃事务，调用 SqlContext::Execute 在事务上下文中执行；如果不存在活跃事务，调用 SqlContext::PollExecuteGeneralSql 在自动事务模式中执行。执行完成后，结果通过 FortTableWriter 格式化并输出。

### 事务状态管理

TZDB Shell 维护一个当前事务指针（current_txn），用于跟踪活跃事务。事务状态机包括以下状态：无事务（current_txn == nullptr）、RUNNING（事务进行中）、COMMITTED（已提交）、ABORTED（已回滚）、TAINTED（被污染）。

状态转移遵循以下规则：从无事务状态执行 BEGIN 命令进入 RUNNING 状态；从 RUNNING 状态执行 COMMIT 命令进入 COMMITTED 状态并返回无事务状态；从 RUNNING 状态执行 ROLLBACK 命令进入 ABORTED 状态并返回无事务状态；在 RUNNING 状态执行失败时进入 TAINTED 状态，必须执行 ROLLBACK 才能返回无事务状态。

### 错误处理机制

TZDB Shell 实现了完整的异常捕获机制。所有 SQL 执行操作都被包装在 try-catch 块中，捕获 std::exception 异常。捕获异常后，系统输出错误信息到标准错误流，并调用 mark_txn_tainted 函数将活跃事务标记为 TAINTED 状态。

在自动事务模式下，失败的查询自动回滚，不影响数据库状态。在显式事务模式下，失败的操作将事务标记为 TAINTED，强制用户显式处理。这种设计确保了错误不会被隐式忽略，用户必须显式确认错误恢复。

### 资源管理与清理

TZDB Shell 采用 RAII（Resource Acquisition Is Initialization）原则管理资源。Database 对象在栈上创建，作用域结束时自动释放。命令行历史记录通过 linenoiseHistoryFree 显式释放。

程序退出时，系统执行以下清理操作：检查是否存在未提交的活跃事务，如存在则自动回滚；释放命令行历史记录；关闭数据库连接。这种设计确保了即使异常退出也不会留下孤立的事务或资源。

## 总结

TZDB Shell 是一个功能完整、设计精良的数据库交互工具。其设计综合考虑了易用性、功能完整性、安全性和可靠性。通过 REPL 交互模型、完整的事务管理、实时状态反馈和清晰的错误处理，TZDB Shell 为用户提供了一个高效、安全的数据库操作平台。系统采用分层架构，清晰分离了用户交互、业务逻辑和数据库引擎层。完善的事务状态管理和错误处理机制确保了数据的一致性和系统的可靠性。该工具在数据库开发、测试、管理和问题诊断中具有重要的应用价值。
