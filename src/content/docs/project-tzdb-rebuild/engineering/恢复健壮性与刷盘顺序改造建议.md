## 目标

这组改造建议分成三个层次:

| 层次 | 目标 |
| --- | --- |
| 短期 | 断电后第二次打开不崩溃 |
| 中期 | 可配置的提交持久性等级 |
| 长期 | 恢复路径与 repair 能力完善 |

## 短期优先项

### 1. 强化 WAL 尾部解析

恢复阶段应把 WAL 当成不可信输入处理。建议在扫描 WAL 时补充以下检查:

| 检查项 | 目的 |
| --- | --- |
| header 长度是否完整 | 防止尾部半写直接越界 |
| 文件剩余字节数是否足够容纳 record | 防止读取截断 record |
| record type 是否合法 | 防止随机字节被解释成有效日志 |
| LSN 是否单调 | 防止尾部垃圾破坏恢复顺序 |
| checksum/CRC 是否通过 | 尽早识别损坏记录 |

一旦发现异常，应采用统一策略:

- 停止继续扫描
- 将异常位置之后的 WAL 视为尾部损坏
- 不再尝试重放
- 返回可诊断的恢复结果，而不是直接崩溃

### 2. 对 page 和 catalog 做防御性校验

建议在以下结构加入显式边界验证:

- page header
- B+ 树节点
- page chain 的前驱后继页号
- catalog 头部和关键字段

最低要求是:

- 遇到非法值时安全失败
- 返回错误码或“需要 repair”的状态
- 不继续做危险的内存访问

## 中期改造

### 1. 引入可配置的 `wal_sync_mode`

建议增加配置项，将当前默认行为和强持久行为区分开。

| 模式 | 语义 |
| --- | --- |
| `off` | 维持当前行为，主要依赖 checkpoint 和 shutdown |
| `commit` | 每次 COMMIT 后强制 WAL `Sync()` |
| `group` | 后续可扩展为组提交 |

### 2. 在 COMMIT 路径支持 `FlushUpTo + Sync`

当前建议的逻辑是:

1. `LogTransactionCommit(txn)` 返回 COMMIT 记录对应的 `lsn`
2. 当 `wal_sync_mode == commit` 时，对该 `lsn` 执行真正的 `FlushUpTo + Sync`

这样做的意义是:

- 天脉环境可以开启强持久模式
- 开发或性能优先环境仍保留当前行为

### 3. 把“WAL 先 durable，再刷 page”做成硬约束

`BufferPoolDisk` 现在已经在刷页前调用 `FlushUpTo(page_lsn)`。下一步应把这条路径强化为 durable 语义:

| 当前 | 目标 |
| --- | --- |
| 刷页前 `FlushUpTo(page_lsn)` | 刷页前 `FlushUpTo(page_lsn, sync=true)` |
| 只保证 WAL 先写文件 | 保证 WAL 先 durable |
| 依赖 shutdown/checkpoint 才强同步 | 在关键刷页点也可配置强同步 |

只有这样，才能更严格地满足:

> 写到数据文件的任何 page，其 `page_lsn` 之前的 WAL 已经 durable。

## 长期改造

### 1. 提供 repair 工具或离线修复路径

正常启动路径不适合承担所有修复职责。更合理的方式是:

- 启动时发现严重损坏，直接返回 `ERR_CORRUPTED`
- 提供独立 repair 工具:
  - 扫描 page
  - 重建 page chain
  - 尝试恢复 catalog
  - 重建索引
  - 保留可解析数据

### 2. 建立故障注入测试

建议加入以下测试类型:

| 测试类型 | 目标 |
| --- | --- |
| WAL 尾部截断 | 验证恢复阶段不会越界或崩溃 |
| page 半写 | 验证页头检查与安全失败 |
| catalog 不一致 | 验证能否返回明确错误 |
| kill -9 / 掉电模拟 | 验证真实断电语义 |
| 多次重复恢复 | 验证恢复幂等性 |

## 推荐实施顺序

1. WAL 尾部长度/合法性/CRC 校验
2. page header 与 catalog 防御性校验
3. `wal_sync_mode` 配置项
4. COMMIT 路径支持 `Sync()`
5. 刷页路径支持 `FlushUpTo(page_lsn, sync=true)`
6. 离线 repair 与故障注入测试

## 结论

如果要尽快缓解天脉断电后的崩溃问题，最优先的不是一上来把所有路径都改成每次 `fsync`，而是先做到两件事:

- 恢复阶段遇到半写数据时不崩溃
- 关键路径上把 `Flush()` 和 `Sync()` 的语义真正区分清楚

之后再根据部署环境决定是否开启更强的 COMMIT 持久性策略。
