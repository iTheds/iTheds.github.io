# 沈阳601通信集成测试报告

## 出差概述

**时间**: 2025年6月16日-7月5日
**目的**: 解决上一阶段插入数据缓慢问题，测试天脉操作系统下数据库UDP网络表现性能

## 测试内容

1. 单条消息增删改查落盘时间，写入内存时间，平均值和极值(16字节-64K)
2. 单条消息网络同步时间，测试10000包数据的平均值及极值(16字节-64K)

数据量要求：16B - 64KB，对极值与2倍阶梯值进行测试

## 测试环境

- **ACoreIDE**: V1.0.0.0 - 试用过期
- **ACoreOSMP**: V1.0.0.1
- **ACoreC**: V1.1.3.4_20240702

---

## **测试报告**

### **网络性能基准分析**

网络测试的三个关键指标具有不同的意义：
- **最小值**：反映环境网络的理想性能上限
- **平均值**：代表网络的实际期望性能
- **最大值**：体现网络的稳定性和容错能力

### **天脉环境 vs Windows 环境对比**

#### **基础网络测试**
- **天脉环境**：最小值 583-1921 μs，平均值 1012-2468 μs
- **Windows 环境**：最小值 0 μs，平均值 64-98 μs

#### **Net Pool 接口性能**
天脉环境的 Net Pool 接口集成了信息调度功能，性能表现为：
- 最小值：1023-1923 μs(32KB)
- 平均值：3853-4044 μs(32KB)

相比之下，Windows 环境：
- Connect 接口：最小值 0 μs，平均值 52-293 μs
- Net Pool：最小值 0 μs，平均值 201-512 μs

### **优化空间分析**

#### **Windows 环境优化潜力**
- Connect 层优化空间：约 20 μs
- Net Pool 优化空间：约 200 μs

#### **天脉环境优化潜力**
优化空间更大，约 500 μs 左右。

### **稳定性分析**

天脉环境的最大值表现：
- **基础测试**：8000-59990 μs(非单机波动较大)，单机环境 17602-18190 μs(相对平滑)
- **Net Pool**：15009-16391 μs，表现更加稳定，可能得益于缓冲区优化

### **优化建议与预期**

#### **Net Pool 优化方案**
1. 内存复用和调度完善
2. 订阅队列模式改为 callback 直接处理
3. 采用默认回应架构，提升数据回传速度

#### **性能提升预期**
- **整体提升空间**：1000-1500 μs
- **最小值**：优化空间有限，预计维持在 1023-1923 μs
- **平均值**：从当前 3853-5339 μs 优化至 1700-2800 μs

### **数据库性能分析**

#### **CRUD 操作特点**
在 Net Pool 提供稳定网络接口的基础上：
- 32KB 和 64KB 数据的性能指标差异仅约 5%
- 大数据量操作主要受数据库内存 IO 影响
- 天脉环境下，有盘模式优于无盘模式

#### **事务处理性能**
- 事务提交占用大量时间
- Insert 操作(100 条数据)：3088-6611 μs
- 单表 vs 多表性能差异显著：64KB 单表 1074 μs vs 32KB 多表 191834 μs

### **综合优化目标**

1. **Net Pool 优化**：平均响应时间降至 1700-2800 μs
2. **单库单表优化**：不同数据量下期望性能 221-1074 μs  
3. **整体数据库性能**：在不考虑分布式调度和多表多库的情况下，优化后平均 INSERT 时间预期 3500-6600 μs

**注**：大数据量场景下的性能保障仍需进一步验证和优化。

## 问题总结

### 测试环境问题
1. 信号量open适配问题
2. 通信IO模型，通知顺序问题
3. 线程数量问题：线程过多导致消息接收线程未开启，且未报异常
4. 内存配置，SQL内存池使用额外的大量堆空间申请
5. UDP可能存在丢包问题 - 已解决，使用的网络为测试网络，更改IP后解决

### 软件问题
1. 删除或更新不存在的数据会发生异常，暂时将更新测试移除
2. 批量插入显示内存不够，单条插入反而内存够用
3. 反复关闭netpool可能造成内存泄漏

## 环境架构

不同节点通过交换机连接。
