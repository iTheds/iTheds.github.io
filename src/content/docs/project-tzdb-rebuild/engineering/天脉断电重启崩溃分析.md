## 问题背景

问题现象是:数据库在天脉操作系统上经历断电后，第二次打开数据库目录时直接崩溃；而同一套代码在当前开发环境中通常能够正常重启。

测试入口较为简单:

```cpp
int ret = dbInit(TZDB_TEST_DATA_DIR, "test_log", 0, 0, 0, 0, false, false);
auto id = tableInit(0);
```

这说明问题更可能出在数据库打开、恢复、元数据加载和页链解析等内部路径，而不是测试代码本身。

## 现象解读

- 第一次在干净环境下运行通常可以成功。
- 断电之后，磁盘上的 WAL、page、catalog 或页链可能处于半写入状态。
- 第二次打开数据库时，会进入恢复和元数据读取路径，这时最容易触发崩溃。

如果同样的数据目录在当前开发系统上能够继续运行，而在天脉系统上稳定崩溃，说明两边的文件系统写回顺序、缓存语义或路径处理行为存在差异，天脉环境把原本潜伏的恢复缺陷放大了。

## 高概率原因

### 1. 恢复路径对损坏结构缺少健壮性检查

数据库打开时会做以下事情:

- 打开数据文件
- 读取元数据和 catalog
- 解析 page chain、B+ 树、freelist 等结构
- 回放 WAL 或 redo log
- 构建 buffer pool

一旦断电发生在以下中间状态:

- WAL 只写了一部分
- page header 已更新但页内容没完整落盘
- catalog 元数据和真实数据页不一致

恢复路径如果默认这些结构总是合法，就可能在解析阶段直接越界、空指针解引用或访问非法页号。

### 2. 文件系统和写回策略差异放大了 page chain 问题

已有经验表明，page chain 一类物理结构如果没有通过 WAL 严格保护，而只是依赖页面本身被刷盘，就会在断电后留下“孤儿页”或断裂链表。

在不同平台上，这种中间状态暴露的概率不同:

- 开发环境可能因为文件系统比较“温和”，表现为少量记录丢失但结构仍可解析。
- 天脉环境更可能把半截状态真实落盘，导致第二次打开时直接读到非法结构。

### 3. 数据目录、路径和权限语义可能与普通平台不同

如果 `TZDB_TEST_DATA_DIR` 在天脉系统上指向的是半持久化目录，或者断电后目录内容出现部分丢失、inode 复用、头部不一致等情况，而 `Database` 内部没有对头部 magic、版本号、文件长度做防御性校验，就会在打开阶段崩溃。

### 4. 静态全局状态问题概率较低

`GlobalState g_state` 属于进程内静态状态。当前问题发生在“断电后重新启动进程”，因此它不是第一怀疑对象。除非编译链或运行时环境在 BSS 初始化上存在异常，否则重点仍应放在磁盘恢复路径。

## 定位建议

### 1. 打开恢复路径详细日志

建议在以下位置增加日志:

- `Database` 打开流程
- WAL 恢复
- catalog 加载
- page chain 重建
- B+ 树节点解析

重点记录:

- page_id / next_page_id / prev_page_id
- catalog 头部字段
- WAL record 长度、类型、LSN
- 遇到异常时的 offset 和错误原因

### 2. 在天脉环境获取崩溃栈

如果能抓到崩溃 backtrace，应优先确认崩点属于哪一层:

- WAL 反序列化
- page header 解析
- catalog 恢复
- iterator 或索引遍历

### 3. 在当前环境模拟不完整刷盘

可以用两类方式验证:

- 写入后不做完整 checkpoint/fsync，直接异常退出
- 手动构造被截断的 WAL 尾部或不完整数据页

目标不是完全复刻天脉环境，而是验证恢复代码能否在半写状态下安全失败，而不是直接崩溃。

## 结论

这个问题更接近“恢复逻辑在半损坏文件上的健壮性不足”，而不是 API 调用方式错误。

更准确地说，当前实现对以下场景处理不够严格:

- WAL 尾部半写
- page header 不一致
- page chain 断裂
- catalog 与数据页状态不一致

在天脉这种更容易暴露真实断电状态的环境里，这些问题会放大为第二次打开直接崩溃。
