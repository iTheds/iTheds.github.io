## 从 `dbInit` / `tableInit` 看潜在问题点

### 1. 数据库恢复逻辑在“损坏文件”场景下不健壮

`dbInit` 只是做了:

```cpp
g_state.db = std::unique_ptr<Database>(new Database(db_path, db_name));
g_state.conn = std::unique_ptr<Connection>(new Connection(*g_state.db));
```

真正干活的，是 `Database` / `Connection` 里面的逻辑:

- 打开磁盘文件
- 读元数据 / catalog / page 链表
- 回放 WAL / redo log
- 构建 buffer pool 等

如果:

- 上一次运行在“写了一半 / WAL 刷了一半 / page 没刷完”时被断电；
- 又没有严格的 `fsync` / `FileSync` 保证写入顺序(比如先写数据，再写元数据，再写 checkpoint)；

那么重启后:

- 某些页内容是旧的
- 某些元数据是新的
- page chain / B+树 / freelist / catalog 结构不一致

在你现在的 OS 上，**可能因为文件系统帮你做了更多序列化和写回保证**，所以这些结构虽然不一致，但不会马上崩——最多丢点记录。  
但在天脉 OS 上，写回顺序更“随缘”，就会出现更极端的不一致，导致:

- 解析 page header 越界
- page chain 断裂访问到非法地址
- iterator / catalog 构建时 dereference 空指针  
  → **直接崩溃**。

也就是说，这更像是“**恢复路径在损坏数据下缺少健壮性检查**”，而不是 gtest 自身代码的问题。

---

### 2. 文件系统 / 缓存语义差异导致的 WAL / page chain 问题

TZDB 里之前就有类似问题(你记得的 page chain 断裂问题):

- 有些“物理结构”的更新(如 `next_page_id`)没有写 WAL，只依赖 page 自己被刷盘；
- 如果崩溃发生在“指针更新完但页还没刷完”之间，恢复后就会出现“孤儿页”。

在不同 OS / 文件系统上，这种“中间状态被落盘”的概率不同:

- 传统桌面 OS 上可能概率比较低，或者只是表现为“丢几条记录”但结构还算合法；
- 在天脉 OS 上(通常是嵌入式/国产 OS，文件系统可能是定制的 YAFFS/JFFS/EXT 变种)，**写入重排序 或 writeback 策略不同**
  ，更容易把“半截状态”持久化下来，导致下一次 `Database` 打开时直接读到非法结构，然后崩。

结合你说的“**断电重启后第二次打开就崩**”，非常符合:

- 第一次运行:正常创建/写入，但没有完全 fsync；
- 断电:文件处在中间状态；
- 第二次运行:`Database` 打开 + 恢复时踩到未考虑到的中间状态，crash。

---

### 3. 目录 / 路径 / 权限在天脉 OS 上的特殊性

`dbInit` 里完全信任:

```cpp
g_state.db = std::unique_ptr<Database>(new Database(db_path, db_name));
```

如果天脉那边:

- `TZDB_TEST_DATA_DIR` 指向的是某种“半持久化”目录(比如 RAM-disk + Journaling 的组合)；
- 断电后目录内容被部分清理 / inode 复用；
- 恢复时发现同名 db 文件一部分还在，一部分丢了；

那么 `Database` 内部如果没对这些异常路径做健壮性判断(例如:头部 magic 不对时应主动返回错误、重建或清理)，而是直接假设结构合法，接着解析结构体，也很容易二次崩溃。

### 4. 静态全局状态在特定 OS 上的初始化/销毁问题(低概率)

`GlobalState g_state;` 是静态全局变量，理论上进程每次启动都会重新 0 初始化，**不会跨重启残留状态**。


- 除非天脉编译链 / loader 对 C++ 静态区有 bug(比如没有清 .bss 段)，否则崩溃基本不在这一层；
- 更大的可能仍然是“**打开磁盘文件时**”那一层。

## 一、先减少“脏/半损坏文件”出现的概率

这些是“预防性”手段:

- **严格的写入顺序 + fsync**
    - 典型做法:
        - 先写 WAL / redo log(附带 LSN）→ `fsync(log)` → 再写数据页 → `fsync(data)` → 最后更新 checkpoint / 元信息。
    - 保证:崩溃时要么
        - 数据没刷，但 WAL 里有 redo，可以重放；
        - 或者 WAL 也没刷完，但 checkpoint 指向的是更旧的、安全的一致点。

- **追加式 / Copy-on-Write 结构**
    - 不在原地改元数据，而是:
        - 写新版本(新的 root page / superblock）；
        - flush 完成后再“指针切换”(比如一个 superblock 指针）。
    - 崩溃时最多有多余垃圾页，不会导致结构自相矛盾。

- **关键结构的多副本 / 校验**
    - superblock、root page、page header 写多份:
        - 带 CRC / checksum；
        - 带 version / generation。
    - 崩溃恢复时选择“**校验通过 + 版本最新**”的那份。

---

## 二、恢复逻辑在“文件已经脏/坏了”时的防御性处理

你现在遇到的是这层:**恢复代码默认文件是“合理的”，一旦读到不合理数据就直接崩**。一般要做:

### 1. 所有关键结构都要做显式校验

典型的几个点:

- **Page header 校验**
    - 检查:
        - magic / type 字段
        - size 是否落在合法范围(例如 `size <= page_size`）
        - slot 数量、offset 不越界
    - 不通过就不要访问该页内部数据，而是标记“坏页”。

- **B+ 树 / 链表结构校验**
    - 检查:
        - `next_page_id` / `prev_page_id` 是否在 `[0, max_page_id)` 范围
        - level / key_count 是否合理
    - 一旦发现“明显不可能”的值，不要继续深挖，直接中止那条链。

- **WAL 条目校验**
    - 每条 record 有:
        - 长度、类型、LSN
        - CRC 校验
    - 扫描 WAL 时遇到任何“长度不对 / CRC 错误 / LSN 倒退”就立刻 **截断后面所有 record**，不要再继续重放。

### 2. 一旦检测到不一致，必须“安全失败”，而不是继续跑

策略是:

- **宁可启动失败/降级启动，也不要崩溃 core dump**:
    - 比如:
        - 启动时发现 catalog / root page 坏掉:
            - 返回“需要修复(repair）”错误码，不提供正常服务；
            - 让上层(比如工具）能提示用户“库已损坏，请执行修复命令/恢复备份”。
    - 或者:
        - 某个索引树坏了，但表数据还在:
            - 启动时 disable 这个索引，仅支持全表扫；
            - 在后台重建索引。

- **局部坏数据的隔离**
    - 某个 page 或某条记录校验失败时:
        - 跳过那条记录或那页；
        - 打日志，后续可以做“逻辑校验 / REPAIR TABLE”。

### 3. 提供独立的“修复/重建工具”

很多数据库都会有类似 `REPAIR TABLE` / `fsck` 的工具，逻辑是:

- 不在正常启动路径里“硬扛各种坏数据”；
- 而是提供一个单独命令:
    - 全面扫描文件结构；
    - 尝试重建:
        - page chain
        - B+树
        - 索引
    - 尽可能保留可读数据，丢弃或标记坏块。

对你这个项目来说，可以考虑:

- 提供一个“**offline repair**”程序，专门在检测到损坏时运行；
- 正常 `dbInit` 遇到严重损坏就返回“需要 repair”，不要继续运行直到崩。

## 1. 现在项目里跟 fsync 相关的核心实现

### 1）统一抽象:`FileSystem::FileSync` / `FileHandle::Sync`

在 `file_system.h` 里:

```cpp
virtual void FileSync(FileHandle &handle) = 0;
```

Linux 实现(`local_file_system.cpp`）:

```cpp
void LocalFileSystem::FileSync(FileHandle &handle) {
  int fd = handle.Cast<UnixFileHandle>().fd;

#if HAVE_FULLFSYNC
  if (::fcntl(fd, F_FULLFSYNC) == 0) {
    return;
  }
#endif

#if HAVE_FDATASYNC
  bool sync_success = ::fdatasync(fd) == 0;
#else
  bool sync_success = ::fsync(fd) == 0;
#endif

  if (!sync_success) {
    if (errno == EIO) throw Exception("fsync failed!");
    throw IOException("Could not fsync file ...");
  }
}
```

也就是说:

- 对“真正的文件句柄”，`Sync()` 底层会调用 `fdatasync` 或 `fsync`，在 macOS 上优先 `F_FULLFSYNC`，是比较**硬**的一种刷盘。

### 2）通用 buffered writer:`BufferedFileWriter::Sync`

`buffered_file_writer.cpp`:

```cpp
void BufferedFileWriter::Flush() {
  if (offset == 0) return;
  fs.Write(*handle, data.get(), offset);
  total_written += offset;
  offset = 0;
}

void BufferedFileWriter::Sync() {
  Flush();
  handle->Sync();
}
```

这里 `Sync()` = 先把缓冲数据写出去(`write`），再调 `FileHandle::Sync()` → 最终落到上面的 `fdatasync/fsync`。

### 3）WAL:`WALManager` 的刷盘策略

`wal_manager.cpp` 里:

- 写单条 WAL record 时:

```cpp
// 直接写大 record 的路径
log_writer_->WriteData(...);
if (shutdown_flag_.load() || force_flush) {
  log_writer_->Sync();   // => Flush + fsync
} else {
  log_writer_->Flush();  // 只 write，不 sync
}
```

- 普通 buffer flush:

```cpp
log_writer_->WriteData(...);
if (shutdown_flag_.load()) {
  log_writer_->Sync();   // 进程退出时会 sync
} else {
  log_writer_->Flush();  // 常规只是 flush
}
```

- 建 checkpoint:

```cpp
CheckpointRecord checkpoint_record(...);
TZDB_RET ret = WriteLogRecord(checkpoint_record, true); // force_flush=true
...
if (log_writer_) {
  log_writer_->Sync();   // 再次 Sync，确保 checkpoint 记录 durable
}
```

再结合 `FlushUpTo(target_lsn)`:只是调用 `FlushLogBuffer()`，在非 shutdown 情况下也是 `Flush()` 而非 `Sync()`。

**含义:**

- 普通事务/记录写入:
    - 默认是“写到 log 文件 + 刷 std::flush 或 buffered write”，但不必每条都 fsync；
    - 强制 flush(`force_flush`）时会把当前 buffer 写出去，并可能 Sync 一次（取决于具体路径）。
- 进程退出（`shutdown_flag_`）和 checkpoint 时会调用 `Sync()`，保证 WAL 至少在这些点上是**持久的**。

## 2. 和“断电重启后第二次打开崩”的关系

    - 只有某些“需要强刷”的调用才传 `flash = true` 触发 `db_file_->Sync()`；
    - Shutdown 时会做一次 Sync。

在你说的天脉 OS 场景下:

- 如果文件系统的 writeback / journal 策略和 Linux/你本机不同，可能会更频繁出现:
    - WAL 半写 / page 半写；
    - 元数据与数据文件不同步；
- 而当前恢复逻辑对“半写”场景的健壮性还不足，就更容易在**第二次打开（恢复过程）**时崩溃。

## 一、只在 COMMIT 时 fsync，可能遇到的问题

这里的前提是:

- 普通 INSERT/UPDATE 只写入 WAL 文件（write/flush），不 fsync；
- 事务 COMMIT 时，对 WAL 做一次 fsync。

### 1. COMMIT 到 fsync 之间的“脆弱窗口”

- 调用流程大致是:
    - 写各种 DML WAL（INSERT/UPDATE…）→
    - 写 COMMIT WAL →
    - 调用 `fsync(fd)`。
- 如果崩溃点在:
    - **COMMIT 写入成功，但 fsync 还没返回**:
        - 可能只有部分记录真正到介质；
        - 也可能整个 COMMIT 记录还没持久化；
- 结果:
    - 对应用来说“COMMIT 已返回成功”；
    - 但从持久化角度看，这个事务可能并没有真正 durable。

> 这属于经典的“**lost commit**”风险:应用以为提交成功，但掉电后恢复时看不到这条事务。

### 2. WAL 尾部出现“半条记录 / 半个 COMMIT”

- DML + COMMIT 这些 record 是顺序 append 写入；
- 如果设备只刷了一部分 block，WAL 文件尾部可能是:
    - 只有 record 头，没有完整 payload；
    - COMMIT record 被截断；
- 如果恢复逻辑简单按“读到 EOF 前所有记录都当合法”来解析，很容易:
    - 读到垃圾长度 / 非法类型；
    - 访问越界，直接崩。

### 3. 数据页与 WAL 顺序不当

即便只在 COMMIT 时 fsync WAL，你还要保证:

- **先 WAL durable，再允许数据页刷盘**；
- 否则可能出现:
    - 数据页已经把修改刷到磁盘；
    - 但对应 WAL / COMMIT 还没 fsync；
    - 崩溃后恢复时，从 WAL 看不到这次事务，只能当成“未提交”，  
      但数据页上已经是“新值”，造成一致性问题（所谓 “no-steal, force”/“steal, no-force” 策略没遵守）。

## 二、工程上应该怎么处理

### 1. 明确持久性语义 + 可配置

- 定义配置，比如:

    - `wal_sync_mode` / `sync_on_commit`:
        - `off`:只在 checkpoint / 关库时 fsync（高性能，弱持久）；
        - `commit`:每个事务 COMMIT 都 fsync 一次 WAL；
        - `group`:按一定时间/批次 group commit，共享一次 fsync。

- 在天脉这种“经常掉电 + 希望不丢 commit”的环境下，可以默认 `commit` 或 `group`。

### 2. 只在 COMMIT 上 fsync 的正确性要点

- **WAL 顺序保证**:
    - 所有 DML WAL（INSERT/UPDATE/DELETE）必须在 COMMIT WAL 之前写入；
    - `fsync` 返回后，至少保证:
        - 这条事务的所有 WAL（含 COMMIT）已经在介质上；
- **数据页写出顺序**:
    - buffer pool 刷页时，必须确保:
        - 对应修改的 WAL 已经 fsync（LSN 顺序）；
    - 常规做法:
        - 每个 page 记录 `page_lsn`；
        - 刷页前确保 `FlushUpTo(page_lsn)` 已完成且 fsync 过。

### 3. 恢复逻辑要对“半写 WAL”健壮

- 扫描 WAL 文件时:
    - 每条 record 带长度 / 校验（checksum/CRC）；
    - 一旦:
        - 读不到完整 header；
        - 或 payload 长度不足；
        - 或 checksum 不匹配；
    - 就把**当前 offset 之后的内容全部视为无效，直接截断**，而不是继续解析。
- 对 COMMIT record:
    - 如果只看到“DML 记录”，而没有看到对应 COMMIT，就当作 loser txn 回滚；
    - 不会因为 “缺个 COMMIT 的尾巴” 把事务误判为已提交。

### 4. 处理“lost commit”的策略

即便你按照“COMMIT → fsync”做，有极短时间窗口是:

- 应用已经拿到“COMMIT 成功”；
- 崩溃发生在内核 / 设备还没真正落盘之前。

这在**任何现实系统里都存在**（除非使用特殊硬件/事务日志设备）。应对方式通常是:

- 在文档中说明:
    - 只有当 `fsync` 成功返回后且不出现严重 I/O 错误时，才保证持久性；
    - 如果底层设备在崩溃/电源故障时丢数据，这是硬件级问题（需要电容/电池保护、RAID cache 等）。
- 在代码里:
    - `fsync` 返回错误（尤其是 `EIO`）要视为**严重错误**，直接让实例进入“需要人工干预/只读”模式；
    - 不能继续提供“看似正常但可能丢数据”的服务。

---

## 三、总结一段话

- **只在 COMMIT 时 fsync WAL** 是一个常见、合理的折中方案，但会带来:
    - “lost commit”的极小时间窗口；
    - WAL 尾部半写、需要恢复逻辑牢靠处理；
    - 必须保证“WAL 先 durable、数据页后刷盘”的顺序。
- 处理方式就是:
    - 明确配置/语义；
    - COMMIT → `FlushUpTo(lsn)` + `fsync`；
    - 刷页前检查 `page_lsn` 是否已经被 fsync；
    - 恢复时对尾部不完整 WAL 做严格校验和截断。

你的现状是:
