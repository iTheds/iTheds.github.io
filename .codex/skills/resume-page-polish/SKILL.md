# resume-page-polish

用于维护 `src/content/docs/personal-resume/` 及其相关组件。目标不是炫技式重构，而是在保持页面稳定可打印的前提下，持续优化内容结构、可读性与复用性。

## 必须遵守的规则

1. 先保排版稳定，再谈组件抽象；若抽象会引入明显布局风险，应优先保留现有可用结构。
2. 不允许为了“代码更优雅”而大规模改动已稳定的页面结构。
3. 每次改动后必须验证页面没有出现明显错位、断行异常、打印分页恶化或卡片高度突变。
4. 页面风格保持正式、干净、低饱和；只允许少量蓝色作为强调色，不引入花哨渐变和过多彩色卡片。
5. 头部信息、联系方式、教育经历、工作经历、项目经历应优先保证信息密度和打印友好，而不是视觉装饰。
6. 对正文内容的修改应最小化；若用户只要求改格式、改布局、改标题，不得改写原有内容表达。
7. 若用户已经确认某种文案格式或展示形式，应优先沿用，不要擅自改成另一种结构化形式。
8. `Skills`、`工作经历`、`教育经历` 等模块的显示形式需保持一致性，但不要强行把所有模块抽成同一种卡片。
9. 打印态是一级约束；新增样式时必须考虑 `@media print` 的表现。
10. 若用户未明确要求，不要提交；默认只修改工作区内容，等用户明确说“提交”再提交。

## 简历页面的默认约束

1. 简历页需要支持直接打印为 PDF。
2. 页面应预留头像位置，默认可使用占位头像，后续可直接替换文件。
3. `基础信息` 应紧凑、正式，优先使用字段名 + 值的形式。
4. `工作经历` 应支持：
   - 单位名称
   - 岗位
   - 任职时间
   - 该单位下的时间线内容
5. `教育经历` 应支持：
   - 学校
   - 专业
   - 学历
   - 时间
   - GPA 展示
6. `Skills` 应聚合技能相关内容，但不能牺牲阅读自然度；若用户偏好普通句式列表，应优先保留句式列表。
7. 组件抽象应服务于复用，而不是制造更多中间层；若某次拆分导致排版风险变高，应回退为更直接的实现。

## Skills 专项约束

1. `Skills.items` 默认沿用工作经历时间线的视觉语言，但不要求强制复用同一组件。
2. `Skills.items` 的内容格式默认写为：
   - `[能力方向]：[关键词 1]、[关键词 2]、[关键词 3]。`
3. `Skills.items` 应优先写“能力方向 + 关键词短语”，不要展开成长句说明，避免和工作经历、项目经历争抢篇幅。
4. 若用户只要求统一样式，不要顺手做高风险的组件抽象；优先通过统一内容格式和局部样式修正达成一致。
5. 打印态下，`Skills.items` 的字号、行高、缩进和 inline code 观感应尽量与工作经历时间线一致。

## 低风险演进策略

1. 优先做数据层统一，例如：
   - 占位符集中管理
   - 类型定义统一
   - 简历页数据对象统一
2. 组件拆分应分阶段进行：
   - 先抽纯内容区块
   - 再抽版式容器
   - 最后再考虑样式拆分
3. 对 `ResumeShell` 这类壳组件的改动要非常保守；若要拆分，先保证外层 class 和样式选择器不变。
4. 改样式时优先局部调节，不先动全局结构。
5. 若用户已经手动回退某次改动，后续不得在同一路径上重复采用同类高风险方案。

## 推荐工作流

1. 明确这次改动是：
   - 内容改动
   - 结构改动
   - 样式改动
   - 打印优化
2. 若涉及结构改动，先检查是否能通过更小范围的修改完成目标。
3. 修改后至少执行一次：
   - `npm run build`
4. 若改动影响打印或布局，优先要求本地预览确认，再继续下一层抽象。

## 不建议做的事

1. 不要一次性重写整个简历组件体系。
2. 不要把用户已经确认的文案格式，擅自转换成表格、双列说明、键值对等其他形式。
3. 不要为“更规范”而引入过多中间组件，导致排查样式困难。
4. 不要在未确认效果前，把多个结构层同时替换。
